# Skill 开发规则

> 定位：本文件用于规范本仓库中 Skill 的设计、编写与优化方式。
> 目标：让 Skill 更容易被触发、更省 token、更稳定，并且输出更像有经验的人写的。

## 1. 先记住这 5 条

1. Skill 只解决一个明确问题，不做大而全的万能助手。
2. `SKILL.md` 只写模型不知道、但对这个场景很关键的信息。
3. 长内容不要堆在 `SKILL.md`，放到 `references/` 按需读取。
4. 规则要尽量可执行，少写口号，多写判断标准。
5. 好 Skill 不只是“内容多”，而是“触发准、动作清、输出稳”。

## 2. 目录规范

每个顶级目录都是一个独立 Skill，彼此之间不应依赖。

### 最小必需结构

```text
{skill-name}/
  └── SKILL.md
```

### 推荐可选结构

```text
{skill-name}/
  ├── SKILL.md
  ├── references/     # 长文档、案例、API、背景材料
  ├── scripts/        # 可直接执行的脚本
  └── assets/         # 图片、图表、素材
```

### 谨慎添加的结构

```text
{skill-name}/
  ├── README.md       # 只有确实需要维护说明时再加
  └── config/         # 只有确实有配置文件时再加
```

规则：
- `SKILL.md` 是唯一必需文件。
- 不要为了“看起来完整”而机械创建空目录。
- 如果 `references/` 存在，`SKILL.md` 里必须明确说明什么时候读取哪个文件。
- 不允许依赖其他 Skill 目录中的文件。

## 3. 什么内容该写，什么不该写

### 应该写的内容

- 这个 Skill 解决什么具体问题。
- 在什么场景下应该触发。
- 输出应该长什么样。
- 模型容易做错的地方。
- 这个领域里特有的规则、约束、风格和禁忌。

### 不该写的内容

- 通用常识，例如“什么是 Markdown”“什么是 Python”。
- 大模型本来就知道的泛化写作建议。
- 没有判断标准的空话，例如“要高质量”“要专业”“要深度思考”。
- 重复三遍本质相同的要求。

### 判断标准

写完每一段后问自己：

> 如果删掉这段，模型还会稳定做对吗？

如果答案是“会”，这段就该删。

## 4. `description` 怎么写

`description` 不是唯一触发因素，但它通常是最关键的触发入口。  
写不好，Skill 很可能根本不会被用到。

### 推荐公式

> [做什么] + [输出什么] + [什么时候用] + [用户可能怎么说]

### 合格的 `description` 应满足

- 说明核心能力，而不是只写领域名。
- 说明输出形式，例如 Markdown、提纲、表格、步骤清单。
- 列出 3 到 5 个典型触发场景。
- 尽量覆盖用户真实会说的话，而不只是概念词。

### 反例

```yaml
description: 帮助写研究报告
```

问题：
- 太短，看不出具体能力。
- 不知道输出形式。
- 不知道什么时候该触发。

### 正例

```yaml
description: |
  生成结构化的深度研究报告，支持技术调研、竞品分析和趋势洞察。
  输出为多章节 Markdown，包含结论、论据、案例和来源标注。
  当用户需要研究某个行业、分析某个产品、撰写调研材料、
  比较多个方案，或明确提到“调研”“分析”“趋势”“竞品”时使用。
```

## 5. `SKILL.md` 推荐骨架

不是每个 Skill 都要完全一样，但建议至少覆盖下面这些部分。

```md
# Skill Name

description: |
  [核心能力] + [输出形式] + [触发场景]

## 何时使用
- 用户提到...
- 用户需要...

## 不适用场景
- 不处理...
- 不负责...

## 核心流程
1. 先判断任务是否属于本 Skill 范围。
2. 明确输出目标和格式。
3. 仅在必要时读取 references 中的指定文件。
4. 按本 Skill 的风格和约束产出结果。

## 输出要求
- 输出必须包含...
- 输出不要出现...

## 风格约束
- 禁止...
- 必须...

## 读取 references 的条件
- 当用户提到 A 时，读取 `references/a.md`
- 当任务涉及 B 时，读取 `references/b.md`
```

## 6. 去 AI 味，不要只写原则，要写成约束

“去 AI 味”不是一句口号，必须写成可执行规则。

### 建议直接写进 Skill 的约束

- 第一段直接进入任务，不要先铺垫背景。
- 禁止使用“在当今”“随着……发展”“综上所述”“总之”等套话。
- 少用“非常”“极其”“显著地”等空泛副词。
- 不要默认套用“首先/其次/最后”的模板结构。
- 每个重要观点尽量用事实、案例、数据或具体动作支撑。

### 反例

```text
在当今快速发展的行业环境中，相关趋势正在发生深刻变化。
首先，我们要理解背景。其次，我们要关注机会。最后，我们要总结价值。
```

### 正例

```text
先给结论：这个方向适合做，但前提是你能拿到稳定的一手案例和可复用的方法论。
如果缺少真实样本，内容会很快变空。
```

## 7. 分层加载怎么做

`SKILL.md` 负责“决策”，`references/` 负责“细节”。

### `SKILL.md` 里应该放

- 使用边界
- 任务判断逻辑
- 核心流程
- 输出约束
- references 的读取索引

### `references/` 里应该放

- 长篇背景资料
- 详细案例
- API 文档
- 大段示例
- 专项说明文档

### 推荐做法

- 超过 500 字的背景说明，优先考虑移到 `references/`。
- 超过 3 个的大样例，优先移到 `references/examples.md`。
- 如果一个知识块不会每次都用到，就不要放在 `SKILL.md`。
- 不要写“按需读取 references”，要写清楚“什么场景读哪个文件”。

## 8. 自由度要匹配任务类型

不同 Skill 适合不同强度的约束，不要一刀切。

### 创意型 Skill

适用：
- 文案
- 策划
- 视觉表达
- 命名和创意生成

写法：
- 给方向、原则、风格边界
- 少给死步骤
- 多给正反例

### 流程型 Skill

适用：
- 调研
- 方案分析
- 复盘
- 文档整理

写法：
- 规定阶段和检查点
- 不必写死每一步的细节
- 重点约束输出结构和质量标准

### 精确型 Skill

适用：
- 数据处理
- 格式转换
- 脚本执行
- 固定流程任务

写法：
- 直接给步骤、命令、判断条件
- 尽量减少模型自行发挥
- 如果有脚本，优先让模型调用脚本而不是自由生成

## 9. 一个 Skill 必须说清楚“不做什么”

很多 Skill 不好用，不是因为能力不够，而是边界不清。

建议显式加入 `不适用场景`，例如：

- 用户问题与本主题无关时，不要强行触发。
- 不要代替其他 Skill 处理跨领域问题。
- 不要在没有明确信号时一次性读取全部 references。
- 不要编造数据、案例或来源。
- 如果任务只需要通用能力，不需要强行套用本 Skill。

## 10. 固定分支优先用选项，不要让用户自由输入

如果某个问题的答案本来就是有限个固定类别，就不要让用户自己组织语言输入，应该直接给选项让用户选。

适用场景：

- 模式选择
- 文档类型选择
- 输入来源选择
- 计划粒度选择
- 平台选择
- 是否开启某个固定功能

规则：

- 能明确列出候选项的，就直接给选项，不要让用户自由输入。
- 固定分支问题优先使用 2 到 5 个选项，避免开放式提问。
- 只有在答案确实开放、需要用户补充业务细节时，才使用自由输入。
- `SKILL.md` 中如果存在固定流程分支，必须写清楚可选项，而不是只写“先询问用户”。

反例：

```text
请告诉我你想用哪种模式。
```

问题：

- 用户不知道有哪些模式可选。
- 用户容易回答成模糊自然语言，导致后续分支不稳定。
- 同一个 Skill 在不同对话里容易走出不一致路径。

正例：

```text
请先选择本次模式：
- 新需求
- 增量需求
```

判断标准：

- 如果用户回答本来可以通过点选完成，就不要设计成自由输入。
- 如果一个分支的合法答案可以提前枚举，就必须枚举出来。
- 选项式提问的目标不是减少交流，而是减少歧义和返工。

## 11. 开发时优先写“规则”，不要写“愿景”

下面这些表达建议少写：

- 输出更专业
- 提升质量
- 更有深度
- 更有人味
- 更像专家

因为它们都缺少动作和判断标准。

更好的写法是：

- 第一段必须直接给出判断或结论。
- 输出中不得出现套话开场。
- 至少提供 2 个具体例子或可执行建议。
- 涉及事实判断时，优先给出依据而不是形容词。

## 12. 开发检查清单

- [ ] 这个 Skill 是否只解决一个明确问题？
- [ ] `SKILL.md` 是否只保留高价值、场景特有的信息？
- [ ] `description` 是否写清了能力、输出和触发场景？
- [ ] 是否写明了不适用场景？
- [ ] 是否明确说明了何时读取哪个 `references/` 文件？
- [ ] 固定分支问题是否优先设计成选项式，而不是自由输入？
- [ ] 是否避免了空洞表达和模板化语气？
- [ ] 如果有脚本，是否优先引导模型调用脚本？
- [ ] 是否做到不依赖其他 Skill 目录？

## 13. 最后标准

一个高质量 Skill，不是“写得多”，而是满足下面四点：

1. 容易被触发。
2. 不容易跑偏。
3. 不浪费上下文。
4. 输出稳定、有信息量、像真人写的。
