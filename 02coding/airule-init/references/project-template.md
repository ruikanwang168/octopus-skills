# PROJECT.md 生成规则

## 文件职责

`PROJECT.md` 是项目快照与技术基准，必须结合真实项目填写项目概述、核心技术栈、核心架构与目录规约、代码规范与红线。

生成 `PROJECT.md` 前必须做源码级深度检索。不能只读取 README、包管理文件或配置文件后就开始写。

源码检索必须先读取并遵守 `AI_IGNORE.md`。如果目标项目尚未创建 `AI_IGNORE.md`，在写入前临时采用 `ai-ignore-template.md` 的默认忽略范围，避免把依赖目录、构建产物、日志、本地环境文件或敏感文件纳入项目事实。

## 源码检索要求

生成或更新 `PROJECT.md` 前，至少完成以下检索：

1. 扫描边界：先读取 `AI_IGNORE.md`，建立忽略列表和允许读取例外；后续检索跳过被忽略路径。
2. 项目结构：使用 `rg --files` 或同等方式列出主要文件，识别源码目录、配置目录、测试目录、脚本目录、构建产物目录。
3. 技术栈证据：读取包管理、构建和运行配置，例如 `package.json`、`pnpm-lock.yaml`、`pom.xml`、`build.gradle`、`pyproject.toml`、`requirements.txt`、`go.mod`、`Cargo.toml`、`Dockerfile`、`docker-compose.yml`。
4. 入口与启动：定位应用入口、路由入口、服务启动类、CLI 入口、主配置文件和环境变量示例。
5. 架构分层：检索控制器、路由、页面、组件、服务、数据访问、模型、状态管理、任务调度、工具函数等目录和命名模式。
6. 数据与外部依赖：检索数据库 schema、迁移脚本、ORM 模型、API 客户端、第三方 SDK、消息队列、缓存、对象存储等集成点。
7. 测试与质量：检索测试目录、测试框架、lint/format/typecheck 脚本、CI 配置和提交检查配置。
8. 代码风格：抽样读取关键模块，不少于 3 类文件；小项目可读完整源码，大项目按模块抽样，确认命名、分层、错误处理和注释风格。

如果项目很大，不要全文读入；先用文件列表和搜索定位关键模块，再抽样读取代表性文件。

## 抽取规则

- 项目概述必须来自真实代码职责，不只复述仓库名。
- 被 `AI_IGNORE.md` 忽略的路径不得作为项目事实来源，除非用户明确要求读取或该任务必须读取；此时需要在摘要中说明原因。
- 技术选型必须带证据来源，例如来自 `package.json`、`pom.xml`、入口文件或配置文件。
- `PROJECT.md` 中的关键结论必须尽量标注来源文件；尤其是技术栈、启动命令、测试命令、数据库、外部服务和核心目录职责。
- 目录说明必须覆盖实际存在的核心目录；不要写不存在的标准目录。
- 如果代码和 README 描述不一致，以代码和配置文件为准，并在 `PROJECT.md` 中标注待确认。
- 对无法确认的信息写“待确认”，不要猜测。
- `PROJECT.md` 正文只保留当前项目事实，不写“旧规则写了什么、已移除什么、这次更新修正了什么”等增量过程说明；这些内容只放在聊天摘要或更新摘要里。

## 默认模板

`PROJECT.md` 必须根据真实项目学习结果填写，不能只输出空标题。

```md
# 项目概览与技术基准

## 一、项目概述

[根据 README、目录结构、包管理文件和入口文件总结项目用途、主要模块和当前开发状态。]

## 二、核心技术栈

[列出真实检测到的语言、框架、构建工具、包管理器、数据库、测试框架、运行环境。每项尽量附来源文件，例如 `package.json`、`pom.xml`、入口文件或配置文件。不要编造未出现的技术。]

## 三、核心架构与目录规约

[说明主要目录职责、模块边界、入口文件、配置文件位置、生成物目录、禁止随意修改的目录。核心目录职责要能追溯到实际文件或目录。]

## 四、代码规范与红线

[写明命名、分层、组件/接口/服务组织方式、测试命令、格式化命令、数据库变更约束、安全红线和已知禁止事项。]
```

## 生成规则

- 每一节都必须有项目特定内容；信息不足时写“待确认”，并列出需要用户确认的具体问题。
- 技术栈只写能从项目文件确认的事实。
- 命令必须来自真实文件，例如 `package.json`、`Makefile`、`pom.xml`、`pyproject.toml`、`README.md`。
- 关键结论缺少来源时，改写为“待确认”，不要当成事实写入。
- 红线优先写会造成破坏、回归或协作断档的约束，不写泛泛建议。
- 更新模式下，先校验旧内容是否仍符合当前项目文件；不能确认的内容标为待确认，不继续当成事实。
- 生成完成前，检查 `PROJECT.md` 的每个结论是否能追溯到实际文件；不能追溯的内容删除或改为待确认。
- 生成完成前，检查 `PROJECT.md` 是否引用了 `AI_IGNORE.md` 中被忽略的路径；如有引用，删除该结论或改为待确认并在聊天摘要中说明。
- 生成完成前，删除正文中的更新过程痕迹，例如“旧规则中的描述已移除”“本次新增”“原来是”等审计说明。
