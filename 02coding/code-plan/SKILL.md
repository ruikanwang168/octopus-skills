# Code Plan

把需求资料与代码证据转换为可分派、可验收的研发计划。保持只读分析；不要修改业务代码、数据库、配置或外部系统。

## 任务契约

### 输入

- 要求提供 PRD 或等价需求基线。无法定位需求基线时停止并要求补充。
- 优先获取页面原型、截图或可运行原型，以及当前代码仓库。
- 接受架构文档、接口文档、数据库说明、研发规范和历史相似需求作为补充证据。
- 原型或代码缺失时允许继续，但必须在总计划中列明证据缺口、无法验证的判断和输出局限。
- 代码可访问时必须检索前端和后端实现；不得仅凭 PRD 设计接口契约。

### 输出

在项目既有文档目录中创建一个需求独立目录；项目没有约定时使用 `docs/development-plan/<feature-name>/`。默认只生成 4 份文件：

```text
00-研发总计划.md
01-前端研发计划.md
02-后端研发计划.md
03-接口契约.md
```

写文件前读取 [references/output-templates.md](references/output-templates.md)，严格使用其中的文档边界：

- 技术方案清单放入总计划，只回答需要产出什么方案、背景是什么、解决什么问题、已有信息是什么；不要撰写方案正文。
- 前端研发计划单独成文；每个任务只设置“计划、前提条件、验收标准”三个正文栏目。
- 后端研发计划单独成文，包含“数据库设计/变更计划”和“后端研发任务”两个章节；数据库任务和后端任务都只设置“计划、前提条件、验收标准”三个正文栏目。
- 数据库设计/变更计划只列需要完成的设计或变更计划，不展开表、字段、索引、约束、关系或 SQL；后端任务必须显式写出数据库依赖链。
- 接口契约明确每个接口的方法与路径、参数和返回值，并给出现有代码证据或待确认标记。
- 总计划只列测试重点、发布依赖、回滚关注点、风险和待确认项；不要生成详细测试用例或完整发布方案。
- 不估算工时，不分配人员，不生成迭代日历。

### 红线

- 不把推测写成事实。无法从 PRD、原型或代码验证的内容标记为“待确认”。
- 不虚构现有模块、文件路径、接口、参数、返回字段或代码规范。
- 不绕过项目已有接口风格另建命名、响应包装、分页或类型规范。
- 不输出 DDL、详细表字段设计、完整 OpenAPI、完整技术方案或实现代码。
- 不用空泛动词充当计划或验收标准，例如“完善”“处理好”“保证正常”。

## 必经检查点

### 1. 建立输入与规则基线

定位 PRD、原型、代码仓库和补充文档。先读取仓库级及相关目录的 `AGENTS.md`、规则文件和文档约定。

建立证据清单，至少记录：

- 需求基线及版本或更新时间；
- 原型页面、路由或截图范围；
- 前端、后端和共享模块的代码根目录；
- 缺失、冲突或无法打开的输入；
- 本次结论允许依赖的事实边界。

只有在 PRD 可定位且证据缺口已显式记录后进入下一检查点。

### 2. 检索现有代码

根据实际技术栈调整检索方式，优先使用 `rg` 和代码导航。至少核对：

- 前端：页面与路由、组件、状态管理、请求封装、接口类型、表单与校验、权限和错误反馈；
- 后端：Controller、Route 或 Handler，Service 或 Use Case，DTO、VO、Schema 或序列化类型，鉴权、校验、响应包装、分页与错误表达；
- 共享约束：命名、日期与枚举格式、ID 类型、空值表达、分页结构、上传下载方式；
- 已有能力：能直接复用、需要修改、需要新增以及疑似废弃的实现。

为重要判断保留文件路径、符号名或路由位置。不要因名称相似就断言可复用；至少核对输入、输出和业务语义。

代码缺失或无法覆盖某一端时，以“证据缺口”退出该项检查，不得伪造规范。

### 3. 建立需求影响映射

把 PRD 条目和原型行为映射到现有页面、模块、接口和数据职责。区分：

- 直接复用；
- 修改扩展；
- 全新增加；
- 删除或替换；
- 无研发变更；
- 待确认。

识别需要单独产出技术方案的事项。出现以下任一情况时列入候选：跨系统或核心模块、引入第三方或新基础设施、复杂状态流转或异步一致性、权限安全、幂等并发或性能风险、存量数据迁移与兼容回滚、现有架构无法直接支撑、存在多个实现方向需要选型。

只创建方案立项清单，不选择或展开具体实现方案。每项严格保留方案名称、背景、要解决的问题和现有信息四部分。

### 4. 生成前端与后端研发计划

按照模板分别生成前端研发计划、后端研发计划。每项任务必须能够独立分派，并仅包含：

- 计划：写明对应需求、影响模块或代码位置、复用/修改/新增/删除方式、主要交付物和任务依赖；
- 前提条件：写明开始任务前必须确定、提供或完成的事项；
- 验收标准：写成可观察、可测试或可从代码与文档中核对的完成条件。

数据库设计/变更计划只列研发前必须完成的数据设计或变更计划，不分析到表、字段、索引、关系或 DDL。若需要详细数据模型设计，把“产出数据库设计”作为数据库计划项或后端任务的前提条件。

数据库与后端的关系必须写清楚：

- 前端研发计划放在 `01-前端研发计划.md`，不得承载数据库或后端实现任务。
- 数据库设计/变更计划和后端研发任务放在同一份 `02-后端研发计划.md` 中，但分为独立章节。
- 凡涉及持久化、查询、保存、发布、删除保护、校验结果承载或数据迁移的后端任务，必须在“前提条件”中引用对应 `DB-xx`，或写明“数据库设计已确认”。
- 后端计划中只有不依赖数据模型的任务可以标注为可并行，例如接口骨架、网关认证接入或通用错误包装；可并行也要写明前提条件。
- 总计划的“研发顺序与依赖”必须能看出技术方案、数据库设计、后端研发、前端联调之间的先后或并行关系。

### 5. 生成接口契约

先以现有前后端代码建立接口规范基线，再逐个接口标注“复用、修改、新增、替换、待确认”。每个接口至少写明：

- 接口名称和业务场景；
- HTTP 方法与路径，或项目实际使用的 RPC、GraphQL 等接口标识；
- 请求参数：位置、名称、类型、必填性、约束和含义；
- 返回值：响应包装、字段路径、类型、可空性和含义；
- 前端与后端代码证据；
- 尚未确定的差异或决策。

已有接口以代码事实为准，并指出 PRD 或原型要求的差异。新增接口遵循仓库现有规范；无法从代码确定的方法、路径、类型或字段必须标记为待确认，不要伪装成已定契约。

### 6. 汇总与一致性检查

初稿完成后读取 [references/review-checklist.md](references/review-checklist.md) 并逐项检查。确保：

- 每条需求进入至少一个计划或明确说明无需研发变更；
- 每个接口能追溯到前端、后端计划或待确认项；
- 所有跨任务依赖在总计划或风险文档中出现；
- 四份文档中的名称、状态、参数和返回值一致；
- PRD、原型和代码冲突没有被静默裁决；
- 验收标准描述结果，不描述含糊过程。

未通过检查时修正文档；证据不足无法修正时，将问题移入待确认项并说明它阻塞哪些任务。

## 模型自主范围

- 根据项目结构选择检索命令、代码入口和证据组织方式。
- 根据需求规模选择按业务流程、页面、领域模块或用户故事组织任务。
- 优先遵循项目既有文档目录和术语；没有约定时使用本 Skill 的默认目录与模板。
- 可以合并重复计划项，但不得因此丢失需求映射、前提条件或验收标准。

## 停止与转交

- 缺少 PRD 或等价需求基线时停止。
- 用户要求直接修改代码、生成 DDL、完成数据库设计、撰写某个完整技术方案或生成详细测试用例时，停止扩张本 Skill 的输出，改用对应专项能力。
- 用户要求执行迁移、发布、权限变更或其他外部操作时，先取得对应流程要求的授权和回滚方案；本 Skill 不执行这些动作。
