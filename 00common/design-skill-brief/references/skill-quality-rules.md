## 五条底线

1. 一个 Skill 只解决一个明确问题，不做大而全的万能助手。
2. `SKILL.md` 只写模型不知道、但对这个场景很关键的信息。
3. 长内容放到 `references/`，并写清读取条件。
4. 所有模式都要写清输入、输出、红线和验收标准。
5. 选择能够控制风险的最低约束级别，让 Skill 触发准、动作清、输出稳。

## 应该写什么

- 这个 Skill 解决的具体问题。
- 什么时候应该触发，什么时候不应该触发。
- 输入需要满足什么条件，输出应该长什么样。
- 哪些红线不能违反，怎样判断结果合格。
- 模型容易做错的地方。
- 只有这个领域才有的规则、约束、风格和禁忌。

## 不该写什么

- 通用常识，例如 Markdown、Python 的基础概念。
- 大模型本来就知道的泛化建议和通用思考步骤。
- 没有判断标准的空话，例如“更专业”“更高质量”“深度思考”。
- 本质相同的要求重复出现。
- 无法说明控制了什么风险、依赖或质量问题的强制步骤。

判断每一段是否保留时，问：删掉这段后，模型还会稳定做对吗？如果会，就删。

## description 规则

`description` 使用这个公式：

```text
[做什么] + [输出什么] + [什么时候用] + [用户可能怎么说]
```

合格标准：

- 说明核心能力，不只写领域名。
- 说明输出形式，例如 Markdown、提纲、表格、步骤清单。
- 列出 3 到 5 个典型触发场景。
- 覆盖用户真实会说的话，而不只写概念词。

不要写成：

```yaml
description: 帮助写研究报告
```

要写成能判断触发的描述：

```yaml
description: |
  生成结构化的深度研究报告，支持技术调研、竞品分析和趋势洞察。
  输出为多章节 Markdown，包含结论、论据、案例和来源标注。
  当用户需要研究某个行业、分析某个产品、撰写调研材料、
  比较多个方案，或明确提到“调研”“分析”“趋势”“竞品”时使用。
```

## SKILL.md 推荐内容

- 使用边界
- 任务判断逻辑
- 输入、输出、红线和验收标准
- 执行约束模式和模型自主范围
- 必要的核心流程
- references 的读取索引

如果背景说明超过 500 字，优先移到 `references/`。如果大样例超过 3 个，优先移到 `references/examples.md`。如果一个知识块不会每次使用，就不要放在 `SKILL.md`。

## 任务契约是所有模式的底座

无论选择哪种执行约束模式，都必须明确：

- 输入：必需信息、可选信息、缺失信息如何处理。
- 输出：形式、必含内容、禁止出现的内容。
- 红线：禁止动作、权限边界、需要用户确认的外部操作。
- 验收标准：可检查的成功条件和失败判定。

红线只说明“不能做什么”，不能替代验收标准；输出格式只说明“长什么样”，不能替代内容质量判断。

## 执行约束模式

执行约束模式回答“模型到达合格结果时需要多少过程约束”。它不是业务领域分类，也不需要同时维护“创意型 / 流程型 / 精确型”这套平行标签。

| 判断维度 | 契约型 | 检查点型 | SOP 型 |
|---|---|---|---|
| 有效路径 | 多种 | 多种，但必须经过关键状态 | 通常只有少数安全路径 |
| 顺序依赖 | 弱 | 阶段间有依赖 | 强 |
| 错误成本 | 低且容易修正 | 中等，需要尽早发现 | 高、不可逆或有外部影响 |
| 结果一致性 | 允许合理差异 | 结构和关键结论需稳定 | 高度一致 |
| 主要约束 | 任务契约和示例 | 任务契约与必经检查点 | 步骤、命令、条件和回滚 |

### 契约型

适用：文案、命名、探索、开放式设计，以及其他存在多种正确路径的任务。

- 给出目标、原则、风格边界、输入、输出、红线和验收标准。
- 多给能体现边界的正反例，少给死步骤。
- 明确模型可以自主决定的内容。
- 不要把“分析需求、开始执行、检查结果”写成没有信息量的步骤。

### 检查点型

适用：调研、需求分析、代码审查、方案设计、复盘等有关键阶段但实现方式可变的任务。

- 规定阶段和必达状态，不写满微观动作。
- 每个检查点写清进入条件、必须产物和退出条件。
- 写清什么情况下暂停、追问、返工或升级为 SOP 型。
- 允许模型自行选择阶段内的工具、顺序和方法。

### SOP 型

适用：数据迁移、格式转换、发布、权限变更、外部消息发送等脆弱或高风险任务。

- 直接给步骤、命令、顺序和判断条件。
- 写明失败处理、重试限制、停止条件和回滚方法。
- 如果有确定性脚本，优先调用脚本而不是自由生成。
- 外部或不可逆动作必须有明确授权条件。

### 混合型

适用：不同阶段的可变性、风险或一致性要求明显不同的任务。

- 为每个阶段单独标注契约型、检查点型或 SOP 型。
- 写清阶段输入、阶段产物和切换条件。
- 只把高风险或强顺序依赖的阶段写成 SOP。
- 不允许用“混合型”掩盖边界不清；阶段无法分清时继续澄清。

## 模式选择规则

按以下顺序判断：

1. 是否存在不可逆操作、外部副作用、安全、权限、合规或高错误成本？存在时优先考虑 SOP 型阶段。
2. 是否存在不能跳过的关键状态或阶段交付物？存在但阶段内方法可变时选择检查点型。
3. 是否有多种有效路径，结果允许合理差异且错误容易修正？满足时选择契约型。
4. 不同阶段的答案不一致时选择混合型，并逐阶段标注。
5. 在能够控制风险的前提下选择约束最少的模式。

模式默认由 Skill 根据证据推荐。只有证据冲突，或用户偏好会改变实际行为时，才提供固定选项让用户决定。推荐结果必须附选择依据，不能只写模式名称。

## 必须写清不做什么

- 用户问题与本主题无关时不要强行触发。
- 不代替其他 Skill 处理跨领域问题。
- 不在没有明确信号时一次性读取全部 references。
- 不编造数据、案例或来源。
- 不在未授权时执行外部或不可逆操作。
- 如果任务只需要通用能力，不强行套用 Skill。

## 固定分支必须用选项

适用于开发方式、输入来源、资源需求、目标位置、交付节奏，以及无法自动判断时的执行约束模式。

规则：

- 能枚举合法答案的，就直接给 2 到 5 个选项。
- 只有业务细节、真实案例、禁忌和特殊要求才开放输入。
- `SKILL.md` 中存在固定流程分支时，必须写清可选项。
- 能从已有信息可靠推断的选项，不重复询问；先展示推断并请用户确认。

不要问：

```text
请告诉我你想用哪种模式。
```

要问：

```text
目前存在两种合理选择，请确认执行约束模式：
- 检查点型：固定必经状态，阶段内允许自主选择方法
- SOP 型：固定步骤、顺序、失败处理和回滚方式
```

## 去 AI 味的可执行约束

- 第一段直接进入任务，不铺垫背景。
- 禁止使用“在当今”“随着……发展”“综上所述”“总之”等套话。
- 少用“非常”“极其”“显著地”等空泛副词。
- 不默认套用“首先/其次/最后”的模板。
- 重要观点用事实、案例、数据或具体动作支撑。

## 开发检查清单

- [ ] 这个 Skill 是否只解决一个明确问题？
- [ ] `SKILL.md` 是否只保留高价值、场景特有的信息？
- [ ] `description` 是否写清能力、输出和触发场景？
- [ ] 是否至少包含 2 个触发说法和 1 个不触发边界？
- [ ] 是否写清输入、输出、红线和验收标准？
- [ ] 执行约束模式是否有证据，并且没有过度约束？
- [ ] 检查点是否描述状态，而不是空泛思考步骤？
- [ ] SOP 是否包含顺序、判断条件、失败处理和回滚？
- [ ] 混合型是否给出逐阶段映射？
- [ ] 是否明确说明何时读取哪个 `references/` 文件？
- [ ] 固定分支问题是否优先设计成选项式？
- [ ] 是否避免并列维护重复的任务类型分类？
- [ ] 是否避免空洞表达和模板化语气？
- [ ] 如果有脚本，是否优先引导模型调用脚本？
- [ ] 是否做到不依赖其他 Skill 目录？

## 最终标准

一个高质量 Skill 应满足四点：

1. 容易被触发。
2. 不容易跑偏。
3. 不浪费上下文。
4. 输出稳定、有信息量，同时保留任务真正需要的自主判断空间。
