# Skill 需求文档模板

读取条件：起草或修改 Skill 需求文档前读取。

## 文档结构

```md
# Skill 需求文档：{推荐 skill-name}

## 结论

- 推荐动作：新建 Skill / 更新现有 Skill / 暂不创建
- 推荐名称：{lowercase-hyphen-name}
- 推荐类型：创意型 / 流程型 / 精确型 / 混合型
- 推荐资源：仅 SKILL.md / references / scripts / assets
- 目标路径：{path}

## 这个 Skill 只解决什么问题

用 1 到 3 句话说明唯一问题。不要把多个不相关能力合并成一个 Skill。

## 触发场景

- 用户可能会说：...
- 用户可能会说：...
- 用户可能会说：...

## 不适用场景

- 不处理：...
- 不负责：...
- 遇到这些情况应改用：...

## 输入与输出

- 用户需要提供：...
- Skill 应输出：...
- 输出必须包含：...
- 输出不得出现：...

## 核心流程

1. ...
2. ...
3. ...

## 固定分支选项

- 模式：...
- 类型：...
- 资源：...
- 位置：...

## references 设计

- `references/{file}.md`：当 ... 时读取。
- 不读取 references 的情况：...

## scripts 设计

- `scripts/{script}.py`：用于 ...
- 输入参数：...
- 输出或退出码：...
- 不需要脚本的原因：...

## assets 设计

- `assets/{name}`：用于 ...
- 不需要 assets 的原因：...

## 容易跑偏的点

- ...
- ...

## 创建前检查清单

- [ ] 只解决一个明确问题。
- [ ] description 写清能力、输出、触发场景和用户说法。
- [ ] 写明不适用场景。
- [ ] 固定分支用选项。
- [ ] references 有明确读取条件。
- [ ] 没有空泛愿景或套话。
- [ ] 不依赖其他 Skill 目录中的文件。

## 待确认问题

- ...
```

## 访谈顺序建议

第一轮只问最影响方向的 3 个问题：

1. 你要新建 Skill，还是更新已有 Skill？
   - 新建 Skill
   - 更新现有 Skill
2. 这个 Skill 主要属于哪类？
   - 创意型
   - 流程型
   - 精确型
   - 混合型
3. 用户说什么话时应该触发它？请给 2 到 3 个真实说法。

第二轮再问边界和输出：

1. 哪些请求不应该触发它？给 1 到 2 个边界例子。
2. 它最终应该输出什么格式？
   - Markdown 文档
   - 表格
   - 文件或目录
   - 脚本执行结果
   - 其他
3. 它是否需要资源？
   - 仅 SKILL.md
   - 需要 references
   - 需要 scripts
   - 需要 assets
   - 暂不确定

第三轮确认落地：

1. 创建位置：
   - 当前工作区
   - `$CODEX_HOME/skills`
   - 用户指定路径
2. 创建节奏：
   - 先确认需求文档，再创建
   - 需求文档足够后自动创建
3. 是否有必须保留的词、风格、禁忌或组织规则？
