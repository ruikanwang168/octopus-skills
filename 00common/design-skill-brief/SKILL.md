# Design Skill Brief

## 目标

把模糊的 Skill 想法整理成一份可执行的 Skill 需求文档，为整个 Skill 或不同阶段选择必要且不过度的执行约束，再把确认后的文档交给 `$skill-creator` 创建或更新 Skill。

## 不适用场景

- 用户已经给出完整、已确认的 Skill 需求文档，只要求直接创建文件时，优先使用 `$skill-creator`。
- 用户只是咨询某个领域知识，不打算创建、更新或评审 Skill 时，不触发本 Skill。
- 不替代 `$skill-creator` 手写最终 Skill，除非用户明确要求在缺少 `$skill-creator` 的环境里继续。
- 不把契约型、检查点型和 SOP 型拆成三个通用 Skill；它们是同一个 Skill 内部的执行约束模式。
- 不一次性读取所有 references；只在下方条件满足时读取指定文件。

## 第一步：校验 skill-creator

在访谈前先确认 `$skill-creator` 是否可用：

1. 先查看当前会话可用 Skills 列表中是否有 `skill-creator`。
2. 如果不确定，运行 `scripts/check_skill_creator.py`。
3. 如果找到，第一轮回复必须说明“已检测到 skill-creator”，然后开始访谈。
4. 如果没找到，停止最终创建流程，并让用户在固定选项中选择：
   - 安装或启用 `skill-creator`
   - 继续只生成 Skill 需求文档
   - 暂停

不要读取 `skill-creator` 目录里的文件来完成本 Skill 的访谈；只有在需求文档确认后，才按 `$skill-creator` 自身规则调用它。

## 访谈规则

- 每轮最多问 3 个问题；信息不足时继续追问，不要一次塞满长问卷。
- 固定分支必须给选项，开放细节才让用户自由输入。
- 优先确认这些固定分支：
  - 开发方式：新建 Skill、更新现有 Skill
  - 交付节奏：只生成需求文档、需求文档确认后创建 Skill、需求足够后自动创建
  - 资源：仅 `SKILL.md`、需要 `references/`、需要 `scripts/`、需要 `assets/`、暂不确定
  - 位置：当前工作区、`$CODEX_HOME/skills`、用户指定路径
- 必须收集至少 2 个应该触发该 Skill 的真实用户说法。
- 必须收集至少 1 个不应该触发该 Skill 的边界例子。
- 必须明确输入、输出、红线和验收标准；可以根据上下文推断，但要把推断作为假设展示给用户确认。
- 需要判断是否存在固定顺序、不可逆操作、外部副作用、权限或合规风险，以及失败后的回滚要求。
- 如果用户已经给出足够信息，先归纳假设并请用户确认，不要重复问已回答的问题。
- 执行约束模式默认由本 Skill 推荐；只有证据冲突，或不同选择会实质改变 Skill 行为时，才让用户从契约型、检查点型、SOP 型、混合型中选择。

## 选择执行约束模式

所有模式都必须先定义任务契约：输入、输出、红线和验收标准。再选择能够控制风险的最低约束级别，不因“写得详细看起来更完整”而增加步骤。

### 契约型

在多种路径都有效、结果允许合理差异、失败容易修正且没有强顺序依赖时使用。

- 规定输入、输出、红线、验收标准和可参考的正反例。
- 写清模型可以自主决定的范围。
- 不规定微观步骤，不要求模型机械复述通用分析过程。

### 检查点型

在过程存在关键阶段或必达状态，但每个阶段的具体方法可以变化时使用。这是复杂流程型 Skill 的默认候选。

- 在任务契约之外，规定必经检查点、进入或退出条件、暂停或升级条件。
- 约束“必须达到什么状态”，不写满“每一步具体怎么做”。
- 允许模型根据上下文选择工具、顺序和实现方式，但不得跳过检查点。

### SOP 型

在操作脆弱、顺序错误会失败、结果必须一致，或涉及不可逆操作、外部副作用、安全、权限、合规和高错误成本时使用。

- 规定步骤、顺序、命令或工具、判断条件、失败处理和回滚方式。
- 能用确定性脚本封装的操作，优先创建或调用脚本，不让模型自由重写。
- 在执行删除、发布、付款、发消息等外部动作前，明确授权或确认条件。

### 混合型

在同一个 Skill 的不同阶段风险和可变性明显不同时使用。

- 给出阶段映射，并为每个阶段单独指定契约型、检查点型或 SOP 型。
- 只对脆弱阶段使用严格 SOP，不把整个 Skill 全局写死。
- 写清阶段切换条件，以及上一个阶段必须交付给下一个阶段的状态或产物。

创意、流程、精确只是判断可变性和风险的线索，不再作为需求文档中的另一套必填分类。无法说明某个步骤控制了什么风险或质量问题时，删除该步骤。

## 需求文档

当信息足够形成文档时：

1. 读取 `references/skill-brief-template.md`。
2. 读取 `references/skill-quality-rules.md`。
3. 输出或写入一份 Markdown 需求文档，文件名优先用 `skill-brief.md`。
4. 写明推荐执行约束模式、选择依据、模型自主范围；混合型必须给出阶段映射。
5. 文档末尾必须包含“创建前检查清单”和“待确认问题”。
6. 写入文件后运行 `python3 scripts/validate_skill_brief.py <skill-brief.md>`，修复全部错误后再交付；只在对话中展示时，按同一检查项人工复核。

需求文档必须直接进入结论，不要用背景铺垫。不要出现“在当今”“随着……发展”“综上所述”“总之”“非常专业”“深度思考”等套话或空泛评价。

## 调用 skill-creator

只有在用户确认需求文档后，才调用 `$skill-creator`。调用时传递以下信息：

- Skill 名称候选和推荐名称
- 目标路径
- 新建或更新方式
- 推荐执行约束模式、选择依据和阶段映射
- 输入、输出、红线和验收标准
- 触发场景和不适用场景
- 必经检查点、严格顺序、失败处理和模型自主范围
- 需要创建的 `references/`、`scripts/`、`assets/`
- 需要遵守的质量规则

如果用户明确授权“文档确认后自动创建”，可以在展示文档后直接进入 `$skill-creator`；否则先问一句是否开始创建。

## 输出要求

- 第一段先给状态或判断，例如是否检测到 `skill-creator`、当前缺少什么信息。
- 问题要短，带编号或选项，避免开放式长段落。
- 需求文档用 Markdown，适合直接交给 `$skill-creator`。
- 每条强制步骤都要能对应具体风险、依赖或验收要求。
- 契约型和检查点型不得填充无意义的通用步骤；SOP 型不得省略顺序、失败处理或回滚条件。
- 涉及事实、路径、脚本、资源目录时给出具体名称，不用“高质量”“更专业”替代判断标准。

## 读取 references 的条件

- 起草或修改 Skill 需求文档前，读取 `references/skill-brief-template.md`。
- 评审 Skill 想法、选择执行约束模式、写创建前检查清单、准备交给 `$skill-creator` 前，读取 `references/skill-quality-rules.md`。
- 仅校验 `skill-creator` 是否存在时，运行 `scripts/check_skill_creator.py`，不需要读取 references。
