# 测试用例设计

## 概览

基于本地产品资料生成测试用例，并尽量延续团队历史用例的格式、粒度和命名习惯。除非用户指定其他路径，否则将当前工作目录视为资料来源。

## 工作流程

1. 发现资料：
   - 使用 `rg --files` 列出文件，并按可能的输入类型分组：历史测试用例、产品需求文档、原型、界面说明、接口文档、数据字典、发版说明和截图。
   - 读取电子表格、文档、PDF、HTML、图片和压缩包时，优先使用结构化读取方式。
   - 如果原型是图片或截图，检查可见状态、控件、文案、导航路径、校验规则，以及空态、异常态和加载态。

2. 学习历史用例规范：
   - 延续主要的产品/模块名称、优先级体系、用例粒度、措辞风格、关键词分类、步骤格式、适用阶段和状态值。
   - 如果历史用例之间存在冲突，优先采用最新或最完整的一组，并在结果说明中标明假设。
   - 优先复用已有回归覆盖模式，再补充新的分类名称。

3. 建立需求映射：
   - 提取业务目标、用户角色、模块、入口、核心流程、业务规则、字段、状态流转、权限、提示信息、外部集成、数据依赖和非功能约束。
   - 为每条需求或每个原型交互至少映射一个正向用例，并基于风险补充异常、边界、权限和回归用例。
   - 当产品复杂、需求文档信息较少，或用户要求大范围覆盖时，读取 `references/coverage-checklist.md`。

4. 设计用例：
   - 除非用户提供其他模板，否则严格使用以下字段和顺序：`所属产品`, `所属模块`, `相关研发需求`, `用例名称`, `前置条件`, `关键词`, `优先级`, `用例类型`, `适用阶段`, `用例状态`, `步骤`, `预期`。
   - 保持用例原子化：每条用例只验证一个行为、规则、状态流转、权限或校验点。
   - 步骤使用编号写成可执行动作。多个检查点并存时，预期结果尽量与步骤编号对应。
   - 能合理推断时避免空值；无前置条件时填写 `无`；不确定内容标记为 `待确认: ...`。

5. 设置优先级和分类：
   - 优先沿用历史命名。如果没有历史规范，使用 `P0` 表示阻塞核心流程、支付/安全/数据丢失风险和发版阻断问题；使用 `P1` 表示重要正常流程和异常流程；使用 `P2` 表示较低风险的界面、兼容性和次要场景。
   - 用例类型使用简洁标签，例如 `功能`, `异常`, `边界`, `权限`, `接口`, `兼容`, `性能`, `安全`, `UI`，或使用项目历史用例中的等价标签。
   - 根据测试目的将 `适用阶段` 默认设置为 `系统测试` 或 `回归测试`；除非历史用例另有默认值，新生成用例的 `用例状态` 默认设置为 `待评审`。

6. 输出并校验：
   - 如果用户未指定格式，同时输出便于阅读的 Markdown 表格/摘要，以及便于编辑的电子表格或 CSV 文件。
   - 为了稳定生成电子表格或 CSV，先将用例写成包含固定字段的 JSON，再运行 `scripts/write_test_cases.py`。
   - 结束前检查每条用例是否有来源需求或明确的推断说明、所有必填字段是否齐全、步骤是否可执行、预期是否可观察。

## 输出脚本

生成文件时使用内置写入脚本：

```bash
python3 /技能路径/test-case-design/scripts/write_test_cases.py 用例.json --output 测试用例.xlsx
```

输入 JSON 可以是用例对象列表，也可以是包含 `cases` 数组的对象。脚本会强制使用固定中文表头、忽略额外字段、保留多行步骤/预期，将字符串里的 `\n`、`\r\n` 和 `<br>` 转成真实单元格换行，并在缺少 XLSX 依赖时自动降级输出 CSV。

## 质量标准

- 在相关场景下覆盖正常流程、分支流程、异常处理、边界值、权限、状态流转、数据持久化、搜索/筛选/排序/分页、导入导出、通知、审计日志、兼容性和集成行为。
- 用例名称要足够具体，让测试人员不读步骤也能区分测试目标。
- 不要把推断当成事实。如果产品需求文档或原型未明确说明，应将推断用例标记为风险覆盖或回归假设。
- 优先追求高价值覆盖，而不是堆数量。合并重复用例，拆分验证无关行为的用例。
