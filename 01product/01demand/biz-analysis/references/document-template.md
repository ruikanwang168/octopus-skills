# 需求梳理文档生成参考

这个文件用于在需求已经基本澄清后，生成正式的 `需求梳理文档.md`。

默认文档是轻量业务梳理稿，服务于客户/业务方确认共识。不要默认写成 PRD、原型说明或开发说明书。

## 生成前检查

生成正式文档前，至少确认或可推断：

- 业务背景是什么。
- 当前核心痛点是什么。
- V1.0 目标或成功标准是什么。
- 准备用什么系统、流程或能力解决。
- 涉及哪些角色。
- 主干业务流程是否能串起来。
- 是否存在外部系统或第三方依赖。

如果缺少业务背景、当前痛点、解决思路、涉及角色、核心流程中的任意一项，不要直接写正式文档，先输出缺口清单。

## 文档标题

使用固定标题：

```md
# 需求梳理文档
```

如用户提供了系统名称，可写成：

```md
# XXX系统需求梳理文档
```

## 默认固定结构

正式文档默认包含以下 8 个一级章节，不要默认增加状态机、字段清单、接口、技术架构等重内容。

```md
# 需求梳理文档

## 一、业务背景

## 二、当前痛点

## 三、业务目标与成功标准

## 四、解决思路

## 五、涉及角色

## 六、核心业务流程

## 七、外部依赖

## 八、待确认问题
```

## 一、业务背景

写清：

- 当前是什么业务场景。
- 谁在什么情况下处理什么事情。
- 现在主要依赖什么方式运转，例如人工、Excel、微信群、纸质单、旧系统等。

要求：

- 用业务语言，不写行业大背景。
- 不要把痛点提前堆在这一章，背景只交代现状。

## 二、当前痛点

写清用户当前遇到的核心问题。

推荐控制在 3 到 5 条，每条包含：

- 痛点是什么。
- 为什么会发生。
- 带来什么影响。

避免：

- 把“想做某功能”直接写成痛点。
- 写“效率低、体验差、管理难”但不说明具体表现。

## 三、业务目标与成功标准

写清这次 V1.0 做到什么算有效。

可以包含：

- 本次上线目标。
- 业务方可感知的变化。
- 可观察的成功标准。

写法示例：

```md
V1.0 目标是让整改事项可追踪、可复核、可统计，减少微信群和 Excel 之间的信息断层。
```

要求：

- 用户给了指标就写指标。
- 用户没给指标时，写可观察结果，不编造数字。
- 如果目标只是当前推测，明确标记“当前推测”。

## 四、解决思路

写清准备如何解决问题，不展开到技术实现。

建议覆盖：

- 用什么系统、流程或工具承接。
- 核心能力有哪些。
- V1.0 做什么，不做什么。

推荐结构：

```md
### 1. 总体思路

### 2. V1.0 范围

| 范围类型 | 内容 | 说明 |
| :--- | :--- | :--- |
| 本期包含 |  |  |
| 本期不包含 |  |  |
| 后续可扩展 |  |  |
```

要求：

- 解决思路要对应痛点，不要罗列无关功能。
- 必须显式写出本期不做什么，防止范围膨胀。

## 五、涉及角色

用表格列出：

```md
| 角色名称 | 参与环节 | 核心职责 |
| :--- | :--- | :--- |
|  |  |  |
```

写法要求：

- 角色名称用业务语言，如“门店店长”“总部运营”“供应商管理员”。
- 参与环节写清角色出现在流程的哪一步。
- 核心职责写到业务动作层面，不写系统权限矩阵。
- 如果数据可见范围是当前核心风险，可以在职责中简要注明；不要默认扩写成权限设计。
- 不写页面权限、按钮权限、菜单权限、字段权限等 UI 或系统配置细节。

## 六、核心业务流程

必须包含主干流程，优先使用 Mermaid 流程图，其次用箭头文本。

推荐写法：

````md
```mermaid
flowchart TD
  A["发起"] --> B["处理"]
  B --> C["审核"]
  C --> D["完成"]
  D --> E["统计"]
```
````

同时补一段文字说明：

```md
主流程：发起 -> 处理 -> 审核 -> 完成 -> 统计。
```

要求：

- 流程图只画主干链路，不把所有异常分支都塞进去。
- 如果有关键分支，例如驳回、退回、取消、逾期提醒，可以用 1 到 3 条补充说明。
- 默认不写状态机表；只有强流转业务或用户明确要求时，才增加“状态说明”小节。

## 七、外部依赖

写清是否依赖外部系统、第三方 API、内部基础数据或人工线下动作。

推荐表格：

```md
| 依赖对象 | 依赖内容 | 当前判断 |
| :--- | :--- | :--- |
|  |  |  |
```

常见依赖：

- 企业微信、钉钉、飞书等消息通知。
- 支付、地图、物流、短信、OCR、AI、文件存储。
- ERP、CRM、OA、财务、人事、组织架构、门店基础数据。
- 线下人工审核、人工录入、人工复核。

如果没有明确外部依赖，写“暂无明确外部系统依赖”；不要强行假设。

## 八、待确认问题

这一章列出需要继续问业务方的问题，不需要写成风险报告。

推荐表格：

```md
| 编号 | 待确认问题 | 为什么要确认 |
| :--- | :--- | :--- |
| Q1 |  |  |
```

要求：

- 问题要具体，能直接拿去问业务方。
- 优先列业务逻辑不闭环、范围边界不清、角色责任不清、外部依赖不确定的点。
- 不要写“需求需进一步确认”这种空话。
- 如果某个问题会影响报价、排期、原型或研发，明确说明。

## 按需扩展

只有在以下情况，才追加扩展章节：

- 用户明确说“给原型用 / 给 PRD 用 / 给研发用”。
- 业务本身强依赖状态流转，例如审批、工单、订单、售后、整改、派单。
- 用户明确要求补字段、状态、权限矩阵、验收标准等。

可追加：

```md
## 附录一、状态说明

## 附录二、核心信息元素
```

扩展规则：

- 状态说明按业务对象拆分，例如“订单状态”“工单状态”，不要把不同对象状态混成一张表。
- 区分状态和计算标记，例如逾期、高风险、异常；计算标记只有触发独立处理流程时才写成状态。
- 核心信息元素只写业务字段，不写数据库类型。

## 风格约束

- 使用清晰、克制的业务分析语言。
- 可以合理推断，但必须标记“当前推测”或“待确认”。
- 不要编造用户没有提供的业务事实、组织结构、系统名称或指标。
- 默认不写接口、数据库表、技术架构细节。
- 默认不写 UI 交互细节，包括页面、按钮、弹窗、筛选、列表列项、表单字段、跳转和提示文案。
- 文档主体先给当前结论，再把不确定内容集中放到第八章。
