# 业务需求梳理

## 核心原则

把用户的表面诉求还原为真实业务问题，再形成覆盖适用考量点的需求基线。模板只负责组织已确认结论，不能代替分析。

本 Skill 输出业务需求基线，不输出完整 PRD、原型、接口、数据库或技术方案。

## 任务契约

### 输入

- 最少需要业务场景、当前问题或希望发生的变化中的一项。
- 可以读取用户提供的需求文档、流程图、原型、截图、操作说明和现状材料。
- 已有材料能证明的内容直接提取，不重复询问；材料只能证明其中明确展示的事实。

### 输出

- 用户确认最终共识后，创建或更新 `需求梳理文档.md`。
- 文档围绕 16 个核心考量点组织，并根据项目类型剪裁；四层分类只用于内部覆盖判断，不作为正式文档标题。
- 必须区分“已确认”“当前推测”“待确认”“不适用”。
- 功能清单保留为业务分析到后续开发任务的桥梁，但不展开页面和技术实现。
- 正式文档以最少文字表达完整业务结论；篇幅不设目标，不为显得完整而扩写或重复。

### 红线

- 不按 16 个考量点逐项提问，不把框架改写成长问卷。
- 不把用户提出的功能或页面直接视为真实需求。
- 不主动追问页面、按钮、弹窗、导航、列表、表单和字段校验。
- 不设计数据库字段、接口、技术状态机、权限配置、日志埋点或监控实现。
- 不编造业务规模、性能数字、组织职责、历史数据规则和外部依赖。
- 不通过互联网搜索推断企业内部规则、系统现状或实际流程。

### 验收标准

- 项目类型已由用户确认。
- 必填项有结论或具体待确认问题；条件项已判断适用性。
- 核心痛点和关键功能已经过真实需求核对与必要性检验。
- 痛点使用 `P1`、`P2` 编号，核心业务能力使用 `C1`、`C2` 编号，功能项填写对应能力编号，形成 `P1 → C1 → 功能项` 的追溯链。
- 每项功能必须关联至少一个能力编号；对应多个编号时使用逗号分隔。
- 目标、范围、角色、流程、功能和防御结论之间没有未暴露的矛盾。
- 每段和每个表格单元格都提供独立信息；删除后不影响业务结论的文字已经删除。
- 用户确认最终共识摘要后才生成正式文档。

## 首次响应

第一次进入本 Skill 时，不直接生成文档，也不抛出完整问题清单。

- 用户已描述系统、场景或问题：先用 3 到 6 条要点复述当前理解，再问 2 到 4 个最影响方向的问题。
- 用户只说“帮我梳理需求”：只问业务场景、主要使用者和当前最痛的问题。
- 用户提供了材料：先提取材料中能确认的结论，再说明关键缺口。

场景基本明确后再确认项目类型，不要在缺少任何业务上下文时机械分类。

## 项目路由检查点

固定类型：

1. 全新产品：0 到 1 新系统、整体重构，或涉及多个角色和完整业务流程。
2. 增量迭代：在正在使用的系统或流程中新增、修改或废弃能力。
3. 小工具：目标单一、角色少、主要链路是输入、处理、输出。

用户未明确类型时，给出当前判断和三个固定选项。用户说“新增、优化、改造、升级”时，优先判断是否存在正在使用的系统或流程，再决定是否属于增量迭代。

推荐表达：

```text
先确认本次需求类型：
1. 全新产品：新建完整系统或整体重构
2. 增量迭代：修改正在使用的系统或流程
3. 小工具：单一目标，主要是输入、处理、输出

当前判断是“增量迭代”，请确认。
```

类型确认后读取 `references/analysis-framework.md`，建立 16 点覆盖表。首次响应和类型确认前不要读取交付模板。

## 16 点覆盖表

四层框架：

- 认知层：名词解释、业务背景。
- 边界层：业务痛点、涉及角色、解决方案、实现边界。
- 交付层：核心流程、功能清单、业务实体、业务状态、数据权限。
- 防御层：异常兜底、外部依赖、老数据兼容、非功能指标、审计可观测。

每项只能标记为：

- 已确认：用户明确提供或确认，没有未处理冲突。
- 当前推测：可以合理推断，但尚未确认。
- 待确认：缺失、冲突或需要业务决策。
- 不适用：按项目类型和当前场景没有适用信号。

覆盖表用于内部判断，不要每轮完整展示给用户。完整适用矩阵、每项分析边界和停止条件见 `references/analysis-framework.md`。

名词解释采用准入制，只收录同时满足以下条件的词：属于当前项目的核心业务领域，代表关键业务对象、规则、流程节点或计算口径；如果不定义，不同角色可能产生会影响范围、流程、规则或结果的理解偏差。普通系统词、页面操作词、通用管理词和语义明确的角色名称不收录；没有通过准入的术语时删除该章节。

## 真实需求检查点

用户提出功能、页面或方案时，不直接记录为需求。先还原：

```text
表面诉求 → 触发场景 → 当前做法 → 真正障碍 → 业务影响 → 期望结果 → 必要能力
```

围绕以下判断深挖：

- 谁在什么情况下遇到问题。
- 当前如何处理，真正卡点在哪里。
- 问题影响谁，会造成什么业务后果。
- 用户希望改变的是结果、规则还是责任关系。
- 提出的能力是否确实是达成目标的必要条件。

对关键功能执行必要性检验：如果不实现它，本期目标是否仍能达成；它解决的是根因还是表面症状；哪个痛点、角色、流程或风险能证明它必要。

根因、业务影响、期望结果和必要能力已经明确后停止深挖。不要机械执行“五个为什么”，不要用重复追问制造深度。

正式记录业务痛点时，先用 1 至 3 句话写总体描述，概括核心矛盾、主要影响对象和整体业务后果，再用 `P1`、`P2` 编号记录具体痛点。

解决方案中的核心业务能力使用 `C1`、`C2` 编号，并填写对应痛点或防御要求编号。功能清单中的每一项填写对应能力编号。没有直接痛点的防御型能力必须关联带编号的异常、外部依赖、老数据兼容或审计要求，不得留空。

## 询问引擎

### 问题准入

只有答案可能改变以下至少一项时才提问：

- 真实痛点或成功结果。
- 本期范围、范围外或保持不变项。
- 角色责任和交接关系。
- 主流程、变化节点或异常处理。
- 功能必要性和业务边界。
- 实体、状态、数据权限的业务定义。
- 外部依赖、老数据、业务量级或审计风险。

如果答案只会决定页面、控件、提示文案或技术实现，不得提问。

### 选择问题

每轮只问 2 到 4 个问题，优先顺序：

1. 会改变目标、范围或主流程方向的冲突。
2. 当前项目类型的必填缺口。
3. 会造成业务中断、数据错误、越权或历史数据风险的问题。
4. 已出现适用信号的条件填写项。

同一优先级内，优先选择能同时推进多个考量点的问题。用户正在连续补充时，先吸收和复述，不抢着追加问题。

### 把原型问题改写成业务问题

- 不问“列表有哪些列”，改问“业务判断必须依赖哪些信息”。
- 不问“按钮放在哪里”，改问“哪个角色在什么时机触发该业务动作”。
- 不问“失败弹窗怎么提示”，改问“失败后业务继续、重试、回滚还是转人工”。
- 不问“数据库有哪些字段”，改问“核心业务实体是什么、有什么关系”。
- 不问“状态如何编码”，改问“什么业务事件触发状态变化、哪些流转不允许”。
- 不问“日志如何实现”，改问“哪些业务事件必须留痕、发生异常时谁需要知道”。

## 项目类型侧重点

### 全新产品

- 深挖真实痛点、完整角色、全景流程、覆盖完整主流程的功能清单和四层防御要求。
- 涉及角色必须结合主流程与用户讨论，确认角色是否真实存在、职责如何划分、是否需要合并或拆分；不得按常见系统角色自行补齐。
- 功能清单按“谁在什么场景下使用什么功能”组织，并说明业务动作、对象和结果。
- 只对通过准入的核心业务领域术语，以及实体、状态、数据权限、非功能和审计形成业务级定义。
- 业务实体只保留有独立业务身份和生命周期的核心对象，不把属性、明细、规则、状态、流程步骤或附件拆成实体。
- 非功能指标优先询问用户量、业务频率、高峰和可接受时效；用户未提供时不编造 QPS。

### 增量迭代

确认类型后，优先请用户提供现状流程或页面材料、本次要改什么、变化发生在哪个节点。材料不要求一次齐全。

- 只梳理变化原因、现状与目标差异、新增/修改/废弃能力和影响面。
- 名词解释只收录本次新增且通过准入的核心业务领域术语，不重复解释既有名词。
- 受影响角色必须与用户讨论并确认，不根据现有页面或常见职责自行决定。
- 功能清单按“谁在什么场景下使用什么功能”组织；修改功能必须标明关联的原有功能和具体变化。
- 业务实体只保留本次涉及的核心实体，控制粒度，并标明新增或修改。
- 业务状态标明既有、新增或修改；只保留理解本次状态变化所需的既有状态。
- 数据权限只写本次相关的新增或修改项。
- 异常兜底、外部依赖和审计可观测只写本次变化直接新增、改变或影响的部分。
- 老数据兼容必须形成结论。
- 重点核对历史数据、在途业务、报表口径、通知、上下游兼容及失败处理。
- 不重复梳理没有变化的全量系统。

### 小工具

- 聚焦业务痛点、解决方案、实现边界、输入、处理、输出、极简动作点和异常兜底。
- 没有适用信号时，不主动追问角色、实体、状态、数据权限、老数据、非功能和审计。
- 实现边界必须说明不能处理的输入、规则限制和不产品化范围。

## 业务定义层边界

- 业务实体：有独立业务身份和生命周期的核心对象、含义及关系；不得按页面区块、表结构或字段明细拆分。
- 业务状态：业务对象、状态、触发事件、流转条件、非法流转和异常去向。
- 数据权限：角色、数据对象、可见范围和业务操作边界。
- 非功能指标：业务量级、容量、时效、并发场景、可用性和保留要求。
- 审计可观测：需要留痕、追踪、统计或告警的业务事件及责任对象。

这些结论可以作为数据库、权限、性能和监控设计的输入，但不得在本 Skill 内完成实现设计。

## 阶段性摘要

每轮关键澄清后输出短摘要，包含：

- 新增的已确认结论。
- 当前判断的真实业务问题，以及它与表面方案是否错位。
- 当前推测。
- 剩余最高优先级缺口。

摘要按业务决策组织，不按 16 个章节复述，不写成正式文档。

## 需求基线检查点

同时满足以下条件，才进入最终共识确认：

1. 项目类型已确认。
2. 当前项目必填项均已确认，或有具体待确认问题及影响。
3. 条件填写项已标记适用、不适用或待判断。
4. 全新产品和增量迭代的涉及角色已经与用户讨论并确认。
5. 核心需求已通过真实需求因果链和必要性检验。
6. 功能清单可以追溯到痛点、角色、流程或防御要求。
7. `P1 → C1 → 功能项` 的结构化追溯链完整，没有未关联的功能项。
8. 主流程能够从触发走到业务结果，关键失败有明确去向。
9. 已确认、当前推测、待确认和不适用没有混写。

以下情况必须暂停生成：

- 真实痛点、本期目标或实现边界未确认。
- 主流程或小工具的输入、处理、输出无法闭环。
- 角色责任存在会改变方案的重大冲突。
- 增量需求的关键老数据或兼容策略不明确。
- 材料相互冲突且尚未确认采用哪个版本。

## 最终共识与文档生成

用户说“可以生成”“开始写文档”时，先展示：

- 项目类型和真实业务问题。
- 目标、解决方案和实现边界。
- 角色、核心流程和功能范围。
- 防御性结论。
- 16 点完整覆盖状态。
- 当前推测和待确认问题。

用户确认后才读取 `references/document-template.md`，按项目类型创建或更新 `需求梳理文档.md`。如果用户刚刚已经确认等价摘要并明确要求生成，不重复确认。

落盘前执行精简复核：删除背景铺陈、同义反复、跨章节重复、空泛评价，以及对表格已有内容的复述。不得以精简为由删除状态标记、追溯编号、范围边界、关键流程、异常处置和风险结论。

正式文档不输出“认知层”“边界层”“交付层”“防御层”。先删除不适用章节，再保持剩余章节的业务顺序，从 1 开始连续重新编号，不允许跳号。显示编号只表示文档阅读顺序；内部 16 点 ID 只用于覆盖判断。

同名文件存在时默认更新原文件；只有用户明确要求保留旧版时才新建版本文件。

文档落盘后，聊天里只说明：

- 已创建或更新的文件。
- 真实问题、目标、边界、主流程和关键防御结论的简短摘要。
- 仍需业务方确认的问题。

## 常见错误

- 按模板章节轮流问问题，导致对话像问卷。
- 为了“深入”重复追问原因，却没有形成业务因果链。
- 用户说要某个页面，就直接补齐按钮、字段和交互。
- 功能清单按常见后台功能凭空扩充，无法追溯到业务结论。
- 把所有条件项都当成必填，导致小工具和增量需求过度分析。
- 业务实体、状态、权限、非功能和审计进入技术实现层。
