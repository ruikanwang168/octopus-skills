# biz-analysis

`biz-analysis` 用于持续澄清业务需求，并最终生成一份结构化的 `需求梳理文档.md`。

适用场景：

- 用户只有模糊业务想法，需要被持续追问。
- 用户要梳理业务系统、内部工具、管理后台、SaaS 模块或流程型系统。
- 用户需要在文档中沉淀业务背景、当前痛点、业务目标、解决思路、涉及角色、核心流程、功能清单、外部依赖和待确认问题。

默认输出：

- `需求梳理文档.md`

文档固定包含：

1. 业务背景
2. 当前痛点
3. 业务目标与成功标准
4. 解决思路
5. 涉及角色
6. 核心业务流程
7. 功能清单
8. 外部依赖
9. 待确认问题

功能清单是正式文档固定章节；状态说明、核心信息元素等内容只在强流转业务，或用户明确说要给原型、PRD、研发继续使用时追加。

澄清阶段默认不问页面、按钮、弹窗、筛选、列表、表单字段等 UI 交互问题，只问业务场景、角色责任、目标、流程、依赖和待确认事项。

功能清单按“角色 / 端 / 功能模块 / 二级功能 / 三级功能 / 业务对象 / 版本范围 / 功能描述 / 功能说明”生成；功能层级只作为业务能力拆分，不展开为页面设计细节。

使用节奏：

1. 先持续提问和复述理解。
2. 信息不足时不直接生成正式文档。
3. 用户明确说“可以生成 / 开始写文档”后，创建或更新 `需求梳理文档.md`。
