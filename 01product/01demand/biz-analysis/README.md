# biz-analysis

`biz-analysis` 用于深挖用户真实业务需求，形成经业务方确认的业务需求基线，再生成 `需求梳理文档.md`。

它不是 PRD、原型或技术方案生成器。产物可以作为后续产品设计、报价、数据库设计和开发任务拆分的输入，但不会主动追问页面、按钮、弹窗、字段、接口或技术实现。

## 四层 16 个核心考量点

| 层级 | 考量点 |
|---|---|
| 认知层 | 名词解释、业务背景 |
| 边界层 | 业务痛点、涉及角色、解决方案、实现边界 |
| 交付层 | 核心流程、功能清单、业务实体、业务状态、数据权限 |
| 防御层 | 异常兜底、外部依赖、老数据兼容、非功能指标、审计可观测 |

16 点是完整度框架，不是固定问卷。Skill 会先提取用户描述和现有材料中的结论，再按项目类型判断必填、条件填写和省略项，每轮只追问 2 到 4 个最影响业务判断的问题。

四层分类和固定的 16 点 ID 只用于 Skill 内部判断。正式文档不输出“认知层、边界层、交付层、防御层”；先删除不适用章节，再按剩余章节的业务顺序从 1 开始连续重新编号，不允许跳号。显示编号只表示文档阅读顺序。

## 三类项目

### 全新产品

- 使用四层完整结构。
- 角色与用户讨论确认；功能清单以“谁在什么场景使用什么功能”为主线。
- 业务实体只保留具有独立身份和生命周期的核心对象，不做细颗粒度拆分。
- 全景流程、状态、数据权限、非功能和审计达到业务定义层。
- 纯新项目没有历史数据时省略老数据兼容。

### 增量迭代

- 名词解释只写新增的核心领域术语；受影响角色与用户讨论确认。
- 功能清单体现角色、场景和功能，修改项关联原有功能。
- 核心实体标明新增或修改；状态标明既有、新增或修改；权限只写新增或修改。
- 异常兜底、外部依赖和审计只写本次相关内容，老数据兼容是重点。
- 不重复梳理未变化的全量系统。

### 小工具

- 聚焦痛点、方案、实现边界、输入、处理、输出、极简功能点和异常兜底。
- 没有适用信号时，不主动深挖角色、实体、状态、权限、非功能和审计。
- 防止把一次性工具扩张成完整产品。

## 真实需求挖掘

用户提出功能、页面或方案时，Skill 不会直接把它写进功能清单，而是先还原：

```text
表面诉求 → 触发场景 → 当前做法 → 真正障碍 → 业务影响 → 期望结果 → 必要能力
```

关键功能还要通过必要性检验：不实现它，本期目标是否仍能达成；它解决的是根因还是表面症状；哪个痛点、角色、流程或风险能证明它必要。

正式文档使用结构化追溯链：业务痛点先写 1 至 3 句话的总体描述，再以 `P1`、`P2` 编号；核心业务能力以 `C1`、`C2` 编号并关联痛点；功能清单填写对应能力编号，形成 `P1 → C1 → 功能项`。一个能力或功能可以关联多个编号。

## 功能清单

功能清单不会取消，但按项目类型使用不同结构：

- 全新产品：按使用角色、使用场景和功能组织完整清单。
- 增量迭代：按角色、场景列新增、修改、废弃功能，修改项关联原有功能。
- 小工具：输入、处理、输出和异常动作点。

功能能力分层不是页面菜单。每项功能必须能追溯到痛点、角色、流程节点或防御要求。

## 使用节奏

1. 理解场景并提取已有证据。
2. 请用户确认全新产品、增量迭代或小工具。
3. 建立 16 点覆盖状态，优先追问矛盾、必填缺口和高风险问题。
4. 每轮用短摘要确认真实问题、新增结论和剩余缺口。
5. 展示最终共识与完整覆盖状态。
6. 用户确认后创建或更新 `需求梳理文档.md`。

正式文档只保留适用于当前项目的内容，不保留空章节。所有结论区分“已确认”“当前推测”“待确认”“不适用”。
