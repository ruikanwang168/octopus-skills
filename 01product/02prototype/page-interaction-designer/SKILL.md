# Page Interaction Designer

## 概览

把已经确认的需求和功能清单，通过“可视化方案 -> 用户点击选择 -> 细节沉淀”的方式，转化为可以交给原型生成、前端实现或视觉设计 skill 使用的页面交互细节文档。

这个 skill 不重新做需求分析，不增删业务范围，不负责配色、品牌视觉或代码实现。它只回答：每个页面怎么组织信息、用户怎么操作、状态如何变化、反馈怎么出现、异常和空状态如何处理。

本 skill 的精髓是：**把页面交互方案画出来，让用户直观看到并选择。** 不要只用文字描述页面交互，除非当前问题确实不适合视觉表达。

## 硬性前提

开始前先确认用户是否已经提供或指向以下输入：

- 已确认的需求说明、PRD、业务流程说明，或明确的功能清单
- 页面范围，或能从需求/代码中推导出的页面范围
- 目标端类型，例如 Web 管理后台、移动端、小程序、桌面端、插件面板

如果缺少需求或功能清单，不要进入交互设计。请用户先完成需求梳理，或建议使用需求分析/PRD 类 skill。

如果只缺少页面清单，但需求和功能清单足够清楚，可以先从需求和现有代码中推导页面清单，并请用户确认。

## 工作边界

负责：

- 读取现有项目结构、页面、路由、组件、文档和设计约束
- 梳理页面清单和页面之间的跳转关系
- 为每个页面确认信息结构、主操作、次操作、状态变化、反馈、空状态、异常状态
- 对页面和复杂交互提出 2-3 种可视化方案，并说明取舍
- 使用浏览器视觉伴侣展示线框图、布局对比、流程图和可点击选项，让用户选择
- 输出页面级交互细节文档

不负责：

- 从零探索业务需求
- 改写 PRD 或重新定义功能范围
- 增加未经用户确认的新功能
- 定义品牌风格、视觉调性、配色、字体或高保真 UI
- 编写、修改或运行产品代码
- 生成最终原型代码

## 流程

必须按顺序执行：

1. **确认输入前提**：确认已有需求、功能清单和目标端类型。若缺失关键输入，停止并说明缺口。
2. **探索项目上下文**：读取当前仓库的文件、文档、路由、页面、组件、设计系统或原型资产；优先使用 `rg` / `rg --files`。如果不是 git 仓库，说明即可。
3. **提取页面清单**：从输入文档和代码结构中整理页面、模块、入口、跳转关系和待确认点。
4. **确认页面范围**：把页面清单发给用户确认。范围不清时，一次只问一个问题。
5. **启动视觉确认环境**：在第一个页面交互决策前，启动本 skill 自带的视觉伴侣服务器。除非运行环境不支持浏览器或用户明确拒绝，否则页面/布局/流程决策都应通过视觉方案确认。
6. **逐页展示可视化方案**：每次聚焦一个页面或一个复杂交互，产出 2-3 个可点击方案。方案可以是线框图、布局卡片、流程图、状态对比或页面跳转图。
7. **读取用户选择并迭代**：读取浏览器点击事件和用户文字反馈。若用户犹豫或要求调整，生成新版本继续确认。
8. **沉淀页面细节**：每确认一个页面，就把用户选择转化为页面交互细节草稿。
9. **汇总细节文档**：把确认后的页面交互细节写成 Markdown 文档。
10. **自查**：检查是否有占位符、需求越界、配色越界、页面遗漏、交互矛盾。
11. **请求用户审阅**：请用户审阅页面交互规则，再交给原型、视觉或开发环节。

## 视觉伴侣是核心流程

本 skill 继承视觉伴侣的核心交互方式：用浏览器展示可点击的交互方案，让用户选择。不要把视觉伴侣当成偶尔使用的附加工具；在页面交互设计阶段，它是默认工作方式。

启动方式：

```bash
scripts/start-server.sh --project-dir /path/to/project --open
```

使用前阅读 `visual-companion.md`。该文件说明如何写可点击方案页面、如何读取用户选择、如何把选择沉淀为交互规则。

启动后保存返回的：

- `url`：发给用户备用
- `screen_dir`：写入 HTML 方案页面
- `state_dir`：读取用户点击事件

如果当前环境无法打开浏览器，或用户明确要求纯文本，才退回文字方案。但最终仍要尽量用结构化 ASCII 图、Mermaid 或表格替代。

必须用视觉方案确认：

- 页面布局 A/B/C 对比
- 复杂表单、详情页、配置页的区域划分
- 多步骤流程、状态流转、条件分支、回退路径
- 页面之间的跳转关系
- 左侧导航、顶部导航、抽屉、弹窗、详情面板等交互模式选择
- 首页、工作台、总览页、配置页、编辑页、详情页等关键页面

可以不用视觉方案确认：

- 需求是否成立
- 功能是否要做
- 字段含义解释
- 技术架构选择
- 配色、品牌调性、高保真风格

如果用户还没有确认需求，不要用视觉草图替代需求澄清。

## 可视化方案页面规则

写入 `screen_dir` 的 HTML 默认使用内容片段，不写完整 `<html>`。每个文件必须使用新文件名，例如：

- `page-list.html`
- `dashboard-layout.html`
- `dashboard-layout-v2.html`
- `node-detail-panel.html`
- `flow-branching.html`

每屏必须包含：

- 当前确认的问题
- 2-3 个方案
- 每个方案的线框图或结构示意
- 每个方案的适用场景和主要风险
- 可点击选项，使用 `data-choice`

示例：

```html
<h2>生命周期总览页用哪种交互结构？</h2>
<p class="subtitle">重点看用户是否能快速理解大节点、小节点、条件分支和回环。</p>

<div class="cards">
  <div class="card" data-choice="map-first" onclick="toggleSelect(this)">
    <div class="card-image">
      <div class="mockup" style="width:92%; margin:0;">
        <div class="mockup-header">方案 A：流程图优先</div>
        <div class="mockup-body">
          <div class="placeholder">大节点横向生命周期 + 小节点流程连线</div>
        </div>
      </div>
    </div>
    <div class="card-body">
      <h3>流程图优先</h3>
      <p>适合强调全局路径、分支和回环，用户先看整体再点节点详情。</p>
    </div>
  </div>

  <div class="card" data-choice="stage-first" onclick="toggleSelect(this)">
    <div class="card-image">
      <div class="mockup" style="width:92%; margin:0;">
        <div class="mockup-header">方案 B：阶段分栏</div>
        <div class="mockup-body">
          <div class="placeholder">按阶段分栏，每栏展示必选/可选小节点</div>
        </div>
      </div>
    </div>
    <div class="card-body">
      <h3>阶段分栏</h3>
      <p>适合节点很多的场景，信息更规整，但条件分支表达较弱。</p>
    </div>
  </div>
</div>
```

用户点击后，下一轮读取 `$STATE_DIR/events`。最后一次点击通常代表当前选择，但也要结合用户文字反馈判断。

如果用户选择不明确，继续出新版本，不要直接写入最终规则。

## 逐页确认方式

每个页面至少确认这些内容：

- **页面目标**：用户来到这个页面要完成什么
- **用户入口**：用户从哪里进入这个页面
- **信息结构**：页面有哪些区域、列表、表单、详情、辅助信息
- **主操作**：页面最重要的操作是什么
- **次操作**：搜索、筛选、排序、导入、导出、复制、更多操作等
- **跳转关系**：会进入哪些页面、弹窗、抽屉、外部链接
- **状态规则**：默认、加载中、已选择、禁用、已完成、进行中、失败等状态
- **反馈规则**：成功、失败、警告、确认、撤销、保存中等反馈
- **空状态和异常**：无数据、无权限、配置缺失、链接失效、校验失败等
- **响应式或端差异**：如果有移动端/桌面端差异，需要明确
- **不负责项**：本页面中哪些内容交给视觉风格、配色、开发或后续需求处理

## 方案比较格式

复杂交互必须给 2-3 种方案，并且优先用浏览器可视化展示。文字里的方案比较只作为浏览器方案的摘要或无浏览器时的降级方案。

```markdown
### 页面：<页面名>

#### 需要确认的问题
<一句话描述当前交互决策>

#### 方案 A：<方案名>
- 交互方式：
- 优点：
- 风险：
- 适合场景：

#### 方案 B：<方案名>
- 交互方式：
- 优点：
- 风险：
- 适合场景：

#### 推荐
推荐 <方案 X>，因为 <结合需求、页面复杂度和现有代码模式说明原因>。

请在浏览器中点击选择 A、B，或告诉我需要调整后再给一版。
```

## 输出文档

默认保存到：

```text
docs/页面交互细节文档/YYYY-MM-DD-<topic>-页面交互细节文档.md
```

如果用户指定了位置，优先使用用户指定位置。

文档必须使用以下结构：

```markdown

- <页面> -> <弹窗/抽屉/外链>：<触发方式>
```

## 自查清单

写完后必须检查：

- 是否真的用可视化方式让用户选择了关键页面和关键交互
- 是否错误地重新做了需求分析
- 是否加入了用户没有确认的新功能
- 是否写了配色、品牌风格、高保真视觉描述
- 每个页面是否都有目标、入口、信息结构、核心交互、状态和异常
- 页面之间的跳转是否闭合
- 弹窗、抽屉、详情页、外链是否说明触发和关闭方式
- 表单是否说明必填、校验、保存、取消、错误反馈
- 是否还有 “TBD”、“TODO”、“待补充” 等占位符

发现问题就直接修复后再交给用户审阅。

## 交接

完成后告诉用户：

- 页面交互细节文档保存在哪里
- 哪些页面或交互仍需要用户确认
- 这个文档可以交给哪些后续环节，例如原型生成、视觉风格/配色 skill、前端实现计划
