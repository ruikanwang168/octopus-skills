## Minimum Inputs

完成“分析已有原型页面并生成 / 维护 Markdown 需求说明”，最低建议提供：

- 处理范围：全量原型代码 / 指定文件 / 指定页面 / 指定功能模块
- 目标文件路径：HTML / Vue / JSX / TSX 或其他原型代码文件
- 输出方式：直接写入 Markdown 说明空间 / dry-run 预览 / patch-only 只给补丁建议 / legacy inline 片段

如果已知，再补：

- 修改意图：新增说明 / 修改既有说明 / 局部重写说明 / 清空说明 / 清空全部说明 / 清空指定页面说明 / 隐藏说明 / 显示说明 / 迁移旧 JSON 或 inline 说明 / 审计说明体系
- 页面类型：列表页 / 详情页 / 表单页 / 工具页
- 页面形态：桌面端 / 移动端 / 大屏 / AIGC
- 页面名称
- 是否需要对子抽屉 / 弹窗 / 子流程单独出说明
- 允许补充的业务上下文
- 是否有必须保留的样式或 DOM 结构
- 页面是否属于 Axure 导出、全屏工作台壳层、iframe/canvas 嵌入页
- 项目是否属于多页面项目：多路由、多 HTML、`portal/admin`、`pages/views`、`router/layouts`
- 当前页面由哪一层负责滚动，是否存在 `overflow-hidden` / 固定侧栏 / 固定底栏
- 是否允许跨文件修改，以及是否允许创建 `prototype-specs/` 或 `src/page-specs/`
- 是否要求后续手动编辑、保存、单页重生成、覆盖确认和历史快照
- 如果是清空模式：是清空全部，还是只清空指定页面 / 指定文件
- 是否需要唯一说明标识：`spec-id`
- 是否需要本轮统一批次标识：`batch-id`
- 是否需要默认展开 / 默认收起 / 用户可手动显隐

若未提供，默认：

- `spec-id = {pageId}-spec`
- `batch-id = spec-batch-YYYYMMDD-HHMMSS`
- 默认显隐 = `visible`
- `storageFormat = markdown`
- `specSchemaVersion = 2`

如果需要更高质量，再补：

- 关键字段定义：字段名 / 组件类型 / 必填 / 默认值 / 校验规则 / 选项来源
- 状态定义：状态名称 / 显示样式 / 生成规则 / 流转条件
- 动作定义：按钮名称 / 触发方式 / 提交规则 / 成功反馈 / 失败反馈
- 关联页面：高级筛选 / 选择器 / 列配置 / 导入导出 / 详情页 / 表单页

## Inference Rules

当输入不完整时，允许保守推断，但要遵守：

- 优先先判定处理范围，再决定读取和写入边界
- 优先从真实页面结构反推模块，而不是从文件名猜业务
- 优先先判定页面形态，再补对应的形态规则
- 优先先判定原型承载壳层和滚动归属，再决定说明区挂载位置
- 优先判断二级承载面是轻量承载还是独立子任务
- 优先补页面中看得见的交互规则，不先补深层业务流程
- 页面中没有的字段、按钮、状态，不要补进说明区
- 无法稳定判断时，写成低假设规则或待补充项，不要伪造复杂流程
- 如果用户点名单页或单功能，默认不扩展到其他页面
- 如果用户没有指定说明区表现形式，默认生成 `current/<pageKey>.md`，而不是把说明直接塞进业务页面
- 如果页面存在 `overflow-hidden`、全屏壳层、Axure 绝对定位画布或 iframe/canvas，默认不要把说明区直接塞进当前可见业务面板
- 如果用户要求清空说明，默认优先删除带显式标记和说明锚点的内容，不要直接删除业务模块
- 如果用户要求后续能精确删除，默认给说明区补 `data-spec-id` 和 `data-spec-batch`
- 如果用户要求后续能临时隐藏，默认补 `data-spec-visible` 和可折叠约定
- 如果项目是多页面项目，或用户说后续还要继续改说明，默认不要生成静态 inline 说明块，优先进入 Markdown-first 可维护说明架构

## Safe Defaults

### 列表页

- 若页面存在筛选区、结果区和分页器，可按列表页处理
- 默认补充查询、重置、刷新、空状态、失败反馈
- 如果存在批量选择控件，再补批量操作规则

### 详情页

- 若页面存在基础信息区、标签区或主操作区，可按详情页处理
- 默认补充打开关闭方式、默认标签、主操作和风险操作规则

### 表单页

- 若页面存在字段输入区和底部提交区，可按表单页处理
- 默认补充字段校验、错误提示、取消和提交

### 工具页

- 若页面存在输入区、处理按钮和结果反馈区，可按工具页处理
- 默认补充处理中、成功、失败三类状态

## Shape Defaults

### 桌面端

- 默认关注标题区、操作区、主体内容区、辅助侧栏、分页器或底部操作区

### 移动端

- 默认关注顶部导航、底部固定操作、卡片列表或分段切换、底部弹层、弱网与空状态
- 若页面存在底部 Tab、吸底按钮或全屏弹层，需要写清显示隐藏和返回路径

### 大屏

- 默认关注指标卡、图表区、地图区、告警区、时间范围、自动刷新或轮播更新
- 若页面存在全屏容器、自动刷新或多块数据联动，需要写清刷新周期和联动关系

### AIGC

- 默认关注输入区、参数区、会话区或结果区、生成中状态、引用来源、取消与重试入口
- 若页面存在多轮会话、版本记录或结果卡片，需要写清生成态、失败态和二次操作规则

## What Not To Infer

以下内容没有输入且页面中不可见时不要强行补：

- 行业专属审批流
- 复杂权限矩阵
- 跨系统同步规则
- 财务、法务、风控等高敏感计算逻辑
- 具体品牌策略或组织内部命名
- 隐藏在后端或模型编排里的复杂推理链路
- 大屏数据口径、传感器接入、模型参数等不可见内部细节

## Preferred Operating Modes

默认优先级从高到低如下：

1. 由执行该 skill 的智能体及其当前模型分析页面并写出高质量 Markdown 正文，特别是单页、少量页面或需要结合用户业务上下文时
2. 使用 `scripts/run_proto_spec_workflow.py` 执行结构化分析、生成/更新、registry/history 维护和可选展示接线
3. 生成或更新 `current/<pageKey>.md`，并维护 `history/` 与 `registry.json`
4. 如用户需要预览，再补最小展示接线
5. 如果用户不允许写文件，输出 patch-only Markdown 内容和建议路径
6. 只有用户明确要求一次性嵌入页面时，才生成 legacy inline 说明块

如果目标是多页面 React / Vue / HTML 项目，且用户预期后续仍会维护说明，则优先级改为：

1. `python3 scripts/run_proto_spec_workflow.py --root <project> --operation create --scope all --integrate` 生成结构化初稿和接线
2. 执行该 skill 的智能体审阅页面代码与生成的 `current/*.md`，补全规则、字段、状态和二级承载面
3. 生成 Markdown-first 说明空间
4. 生成可插入的 loader / 展示层 / 保存层补丁
5. 仅在用户明确接受一次性交付时，才退回静态 inline 说明块

runner 是本地进程，不会自动继承执行环境中的当前模型。不要为了普通 skill 使用要求用户配置模型 API；只有无人值守批量生成或用户明确要求脚本自己调用模型时，才配置 `--model-base-url --model-api-key --model` 或 `PROTO_SPEC_MODEL_*`。

除非用户明确要求，不要把“分析已有页面补说明”退化成“重新生成一个新页面”。
除非用户明确要求，不要把说明区伪装成页面里的普通业务模块；默认给说明区独立 wrapper、标题和最小隔离样式。
除非用户明确要求，不要把说明区做成覆盖式浮层；默认优先选择不会挡住原型的正文后置、外层滚动或安全说明带方案。
如果任务是清空说明，默认优先用 [cleanup-mode.md](cleanup-mode.md) 里的清理脚本或同等的可识别标记策略处理。
如果任务是隐藏或重新显示说明，默认优先用 [visibility-mode.md](visibility-mode.md) 的标记协议和显隐脚本处理。

## Scope Safety Rules

如果任务是全量模式：

- 仅处理用户明确指定的页面集合、目录或原型包
- 不要把未纳入范围的文件默认为可写入目标

如果任务是指定范围模式：

- 只读取和修改被点名的文件或明确依赖文件
- 不要改动其他功能页面
- 如果说明修改依赖共享组件，默认输出局部补丁建议，而不是直接修改共享层
- 只有用户明确允许时，才扩大到跨页面或跨文件修改
