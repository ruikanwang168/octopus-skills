# Strict Interview Mode

在以下情况下，先进入访谈模式，再开始改代码：

- 用户只输入了 skill 名称
- 指令没有明确这次要做什么，或没有明确处理范围
- 用户说“处理整个项目”，但没有说明是否就是全部页面
- 用户提到后续还要手动编辑 / 重生成 / 覆盖确认，但没有明确是否接受可持续维护方案
- 用户只说“帮我处理需求说明”，还无法判断是新增、清空还是只改显示

## Interaction Principles

访谈模式不是问卷模式。默认按下面的顺序做：

- 先自动判断框架、布局、页面结构和可推导范围，不把内部实现术语直接抛给用户
- 首轮优先确认 3 件事：`这次要做什么`、`处理范围`、`后续是否还要继续编辑说明`
- 一次只补问 1 个真正阻塞执行的问题，不要把 5 个问题一起砸给用户
- 高风险操作再确认：全量清空、多文件联动、纯 HTML 的页面内保存
- 不要求用户先选择 `应用壳 / 文档流`、`脚本部署`、`工程模式 / 极速模式` 这类内部概念

## Question Strategy

默认使用“分层确认”，不要走两个极端：

- 不要把所有问题拆成一问一答到底，避免交互过慢
- 不要首轮把所有潜在问题一次性全砸给用户，避免用户漏答和认知负担过高
- 先一次性确认少量高层决策，再按分支逐个补问真正阻塞的问题

### Layer 1: 首轮合并确认

首轮允许一次性确认 2 到 3 个问题，但这些问题必须同时满足：

- 用户能直接回答，不需要理解内部实现术语
- 彼此基本独立，不依赖前一个答案才能理解
- 会直接决定本次执行分支、写入边界或交付形态
- 当前无法从仓库结构、文件路径或已有描述中稳定推断

本 skill 首轮默认优先合并问这 3 件事：

1. 这次要做什么
2. 处理范围
3. 后续是否还要继续编辑说明 / 单页重生成

如果其中某一项已经能稳定推断，就不要重复问；首轮问题数可以降到 1 到 2 个。

### Layer 2: 分支补问

完成首轮后，只在真正阻塞执行时补问，而且一次只问 1 个问题。

适合放到第二轮或后续再问的问题：

- 依赖前一个答案才能成立的问题
- 只对某个分支生效的实现细节
- 高风险操作的最终确认
- 纯 HTML 页面内保存方式
- 清空说明前是否先 `dry-run`
- “只改显示”时具体是默认隐藏、增加切换按钮，还是双视图

### Ask / Infer Decision Rule

每准备问一个问题前，先过这 4 步：

1. 这个答案能不能从仓库结构、文件路径、已有页面名或用户原话里稳定推断？能推断就不要问。
2. 这个答案会不会改变执行分支、修改范围或交付方式？不会改变就不要问。
3. 这个问题是不是用户视角、结果导向的说法？如果还是内部术语，先翻译再问。
4. 这个问题是不是当前唯一阻塞项？如果不是，继续往后执行，不要提前追问。

### What To Batch vs What To Defer

适合首轮合并问：

- 新增 / 修改 / 删除 / 只调显示
- 单页 / 指定范围 / 全部页面
- 后续是否还要继续编辑 / 保存 / 单页重生成

不适合首轮合并问：

- 说明存储层怎么落
- 用哪种骨架
- 页面属于应用壳还是文档流
- 是否部署脚本
- HTML 项目用本地服务还是 `localStorage`
- 清空说明时具体删哪些标记

### Tone Rule

提问时优先用用户目标来表达，不要把 skill 内部分类直接甩给用户。

优先这样说：

- “说明内容后面还要不要继续编辑？”
- “这次只做查看版，还是后面还要改单页说明？”
- “说明先隐藏，还是要加展开/收起？”

不要这样说：

- “你要不要工程模式？”
- “你选极速模式还是可编辑架构？”
- “你要 display-only 还是 editable storage mode？”

## First-Turn Prompt

默认先用这段自然语言引导：

> 我会基于你现有的原型代码处理，不会重做页面。  
> 先确认三点：  
> 1. 这次要做什么：新增说明、修改说明、删除说明，还是只调整说明的展示方式？  
> 2. 范围是：当前页 / 当前文件、指定页面或模块，还是全部页面？  
> 3. 这批说明后面还要不要继续编辑或单页重生成？如果要，我会直接按可持续维护的方式处理。  
> 如果你已经给了文件路径或页面名，我会默认按那个范围处理；框架和布局我会先自动识别。

推荐的自然语言例子：

- 给 `src/pages/market.tsx` 新增说明
- 清空 `ask-data.html` 的说明
- 保留说明，但默认隐藏整个项目里的说明
- 两页都补说明，后面我还要继续改单页内容

## Follow-Up Rules

### A. 如果用户要“新增/更新说明”

不要马上追问“工程模式 / 极速模式”。先按下面的逻辑处理：

- 如果只是单页、一次性评审、没有提后续保存或重生成，默认按“一次性评审版”推进
- 如果已识别为多页面项目，或用户明确提到后续还要编辑、保存、单页重生成、覆盖确认，默认告诉用户将按“可持续维护版”处理
- 只有当这个判断会显著影响交付边界时，才补问一句确认

推荐补问文案：

> 这次生成完之后，你后面还要不要继续改这些需求说明？  
> - 不用了，这次主要是产出一个可查看版本  
> - 要，后面还会手动编辑、单页重生成，或者覆盖确认
如果需要进一步翻译交付差异，再补一句：

> 我会把这次结果做成哪种形式：  
> - 一次性评审版：生成后主要用于查看  
> - 可持续维护版：后续还要编辑、保存、单页重生成、覆盖确认

### B. 如果用户要“清空说明”

不要先问一长串参数。默认按下面的顺序：

- 如果用户已经给了页面、文件、模块或目录，直接按该范围处理
- 如果用户说“全部清空”，默认理解为当前目标项目下的全部说明
- 默认先 `dry-run` 预览，再正式删除
- 只有当范围不清楚时，才补问“是当前页、指定范围，还是全部页面”

### C. 如果用户要“保留说明，只改显示”

这是指说明内容不变，只调整它怎么显示。适用场景：

- 保留说明，但默认先隐藏
- 把默认展开改成默认收起
- 增加“展开 / 收起”按钮
- 对工作台 / 后台壳层页面，改成“产品页面 / 需求说明”双视图

默认可选显示方式：

- 默认显示
- 默认隐藏
- 用户可手动展开 / 收起
- 产品页面 / 需求说明双视图

如果用户意图已经很清楚，例如“先隐藏说明”或“加个展开按钮”，直接执行，不必把 4 个选项全列出来。

### D. 如果是“纯 HTML + 可持续维护版 + 页面内编辑保存”

这是唯一需要额外明确保存方式的常见场景。

推荐补问文案：

> 检测到这是纯 HTML 项目。要支持“页面里编辑并保存”，还需要确认保存方式：  
> - 本地保存服务：页面里编辑后可直接写回项目文件  
> - 仅当前浏览器保存：`localStorage + 导入/导出 JSON`  
> - 不做页面内保存：只展示说明，或手工改 JSON

如果用户没有明确回答，不要把静态 HTML 或纯展示方案误报成“页面内可编辑并保存”。

## Fast Path

如果仓库结构或用户输入已经足够清晰，直接推断，不重复追问：

- 已点名文件路径：默认按当前文件 / 当前页处理
- 已点名多个文件或目录：默认按指定范围处理
- 明确说“全部页面”或“整个项目”：默认按全量处理
- 明确说“先隐藏”“重新显示”“加切换按钮”：默认归入“保留说明，只改显示”
- 明确说“后面还要再改说明”“需要保存”“后续会重生成”：默认归入“可持续维护版”
- 已识别到 `h-screen`、`overflow-hidden`、侧栏壳层：默认优先考虑“产品页面 / 需求说明”双视图
- 只剩 1 个阻塞点时，只问那 1 个问题

## Hard Stop

命中以下任一情况，不要直接盲改代码：

- 不知道目标文件或允许修改的范围
- 不知道用户是要新增/更新、清空，还是只改显示
- 不知道这次是否只是生成查看版，还是后续还要继续编辑 / 单页重生成
- 要做全量清空或大范围改动，但用户没有确认范围
- 纯 HTML 项目要求“页面内编辑并保存”，但保存方式还没确定
- 用户没有提供现成页面或代码文件，却要求直接补页面内说明
