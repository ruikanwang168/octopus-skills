# Quality Checklist

输出前逐项核对：

- 是否先分析了已有页面代码，而不是直接按想象写说明
- 是否明确识别了处理范围：全量或指定范围
- 是否明确选择了一个主页面类型
- 是否明确选择了一个主页面形态
- 是否先识别了页面展示模式：普通文档流 / 固定视口壳层 / 绝对定位画布 / 强覆盖层页面
- 是否先识别了滚动归属，而不是把说明区挂进裁切层
- 需求说明区是否直接对应当前页面中的真实模块
- 页面中的主要元素是否都在说明区中找到对应规则
- 说明区里是否出现了页面上不存在的字段、按钮、状态或弹层
- 是否写清默认值、校验、反馈和边界情况
- 是否用 `15-dimension-scan.md` 做过按需维度扫描
- 15 维中页面可见或用户已补充的维度是否已映射到对应模块规则
- 是否避免把不可见的权限、通知、审计、第三方集成、落库规则写成事实
- 如果用户要求风险/缺口，是否把不可见但可能需要业务补充的维度写入 `待补充维度`
- 页面形态特有模块是否被覆盖
- 是否覆盖空状态、失败分支、禁用态或无权限态
- 是否避免无依据的行业规则和复杂流程
- 如果修改了原文件，是否尽量保持原页面结构和样式不被破坏
- 如果修改了原文件，说明区是否处在正常文档流中，而不是 absolute / fixed overlay
- 如果页面存在 `#base`、大量绝对定位、`h-screen`、`overflow-hidden` 或固定底栏，是否已经针对宿主结构做了安全挂载处理
- 如果目标是多页面项目，是否先判断了它应该进入“可编辑说明架构”，而不是直接退回静态 inline 说明
- 如果目标是多页面或长期维护项目，是否默认生成 `current/<pageKey>.md`，而不是旧 inline 或 JSON 主协议
- `current/<pageKey>.md` 是否包含 `specSchemaVersion: 2`、`storageFormat: "markdown"` 和完整 frontmatter
- Markdown frontmatter 中的 `pageKey` 是否与文件名一致
- `registry.json` 是否包含所有 current Markdown 文件
- `history/<pageKey>/` 是否已为每个 current 页面准备
- 如果页面是 Axure 导出页，说明区是否已经避开 `#base` 现有绝对定位区域，或明确进入安全说明带模式
- 如果页面是 iframe / canvas / svg 承载页，说明区是否挂在其外层，而不是覆盖在承载面上方
- 如果修改了原文件，说明区是否与原型主界面保持了明确视觉隔断，而不是混成同一种业务卡片
- 说明区是否使用了独立 wrapper / class 命名空间，避免复用页面现有的业务样式表达
- 是否先枚举了页面中的全部二级承载面，再决定哪些需要升格
- 如果页面存在复杂二级承载面，是否已单独生成子页面说明
- 如果任务是指定范围模式，是否确认未修改其他无关页面
- 如果涉及共享组件，是否避免了会影响其他页面的联动改动
- 如果任务是清空说明，是否只删除了说明区 / 说明包装器 / 说明 helper，而没有误删业务模块
- 如果任务是清空说明，是否确认已清掉目标范围内的 `PROTO_SPEC` 标记、`data-role="page-spec"`、`withPageSpec` 或其他说明锚点
- 如果任务是清空说明，是否确认页面默认导出、必要 import 和主体渲染仍然成立
- 如果任务是隐藏或显示说明，是否确认 `data-spec-visible`、`hidden` 和显隐按钮状态一致
- 如果任务使用了 `spec-id` / `batch-id`，是否确认这些标识在目标范围内唯一且可检索
- 如果目标是多页面 React / Vue / HTML 项目，真实业务页是否已经接入说明系统，而不是只在 demo 页面里可编辑
- 如果宣称“支持编辑”，是否同时具备编辑入口、Markdown/预览切换、保存链路和持久说明来源，而不是只有静态文本
- 如果存在“预览”按钮，是否确认它渲染当前未保存草稿；只有“保存”会写回 `current/<pageKey>.md` 并生成历史快照
- 如果目标是多页面持续维护场景，是否已经具备当前说明文件层、历史快照位点和覆盖确认检查位点
- 如果目标是可编辑说明架构，`current/*.{md,json}` 是否全部能被校验脚本成功解析；其中 Markdown 是默认主协议，JSON 只做兼容
- 如果目标是可编辑说明架构，是否确认每页 `summary` 非空、每个章节都有非空 `rules`
- 如果目标是可编辑说明架构，真实业务页冷启动时是否真的从 `current/*.md`、由 Markdown 派生的 manifest、或等价持久源读取说明，而不是继续读空 `pages.ts` / `initialSections`
- 如果目标项目里存在 localStorage / store 缓存，是否确认它不是把空章节配置当作第一版说明写回去
- 如果结果要发给其他人继续查看或维护，是否确认编辑保存会真正写回 `current/<pageKey>.md` 或明确标注为 browser-only fallback
- 如果页面切到“需求说明”视图，是否能实际看到非空规则正文，而不只是章节标题
- 如果同一 skill 输出 HTML / React / Vue 三类接线，是否确认三者使用同一套 `.proto-spec-*` 文档阅读器样式、相同按钮文案和相同编辑状态语义
- 真实页面传给说明系统的 `pageKey` 是否与 `current/<pageKey>.md` 文件名一致，而不是由原始 `pathname.replace(...)` 直接拼出来
- 如果说明组件带有“加载需求说明...”这类占位态，是否确认成功和失败两条路径都能退出 loading
- 如果宣称“支持手动编辑”，`section.rules` 是否仍保持结构化编辑，而不是退化成 `rules.join('\n')` 的整段文本框
- 如果宣称“支持手动编辑”，是否实际提供了逐条规则的新增 / 删除能力
- 如果宣称“支持手动编辑”，是否实际提供了新增章节，以及字段表的新增 / 删除能力
- 是否完全避免项目名称、品牌名、来源文件和原文泄露
- 如果输出的是可分享提示词，是否也保持了“先分析现有页面，再补说明”的前提
- 如果页面属于工作台、向导、报告、地图等特殊形态，是否补了对应的额外规则
