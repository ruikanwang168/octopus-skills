这个文件用于约束上游原型生成，使产出的页面代码能够被 `prototype-spec-annotator` 稳定分析并补充页面内需求说明区。

定位说明：

- 它不是后处理时的核心规则文档
- 它是上游生成原型时的兼容性约束模板
- 使用方式是把对应版本追加到 vibe coding 的原型生成提示词末尾

## Drawer And Modal Rule

如果页面中存在抽屉、弹窗、全屏侧滑层或底部弹层，请额外遵守以下约束：

```text
如果页面中存在抽屉、弹窗、全屏侧滑层或底部弹层，请遵守以下约束：

1. 先枚举页面中的全部二级承载面，不要遗漏。
2. 每个二级承载面都要保留独立标题和可识别入口。
3. 如果某个二级承载面包含独立列表、表单、筛选、空状态、返回路径、步骤切换或继续进入下一层流程，请将其按独立子任务来生成结构，不要压缩成普通弹层。
4. 同一页面里如果有多个复杂抽屉或弹窗，必须全部保留独立结构，不能只保留其中一个。
5. 字段名、按钮文案、状态文案、弹层标题必须直接出现在页面模板中，不要只藏在配置对象里。
6. 正式需求说明不会放在蒙版区域；后续将统一在页面下方生成说明区，并通过独立标题、分隔带或文档容器与原型主界面明显隔开；复杂抽屉会作为下方说明区里的独立章节。
7. 不要把正式需求说明预留成固定底栏、右侧覆盖面板或蒙版浮层；如果需要预留说明挂载位，应预留在正文流末尾、外层滚动容器或画布安全带中。
```

## 1. 通用版

适合桌面端后台、业务系统、运营系统、通用 B 端页面。

```text
为兼容 prototype-spec-annotator，请遵守以下约束：

1. 输出页面级原型代码文件，不要只输出零散组件。
2. 每个文件只表达一个主页面类型：列表页 / 详情页 / 表单页 / 工具页。
3. 每个文件只表达一个主页面形态：桌面端 / 移动端 / 大屏 / AIGC。
4. 页面文件中必须保留完整结构骨架，不要把主要结构完全下沉到不可见子组件、纯配置对象或深层渲染函数中。
5. 页面中的关键模块必须可被直接识别，包括：标题区、操作区、主要内容区、状态区、按钮、弹层、分页器、表单区、结果区。
6. 字段名、按钮文案、状态文案、分栏标题、弹层标题应直接出现在 HTML / Vue template / JSX / TSX 中，不要只藏在抽象配置里。
7. 优先使用清晰的模块命名，例如：header、toolbar、filters、results、table、form、tabs、drawer、dialog、footer-actions，或使用 data-module 标记。
8. 如果使用组件化写法，页面文件本身仍需保留主结构、模块标题和主要交互入口。
9. 如果有复杂业务规则但页面中不可见，不要伪造实现；只保留可见结构，后续由需求说明补充。
10. 页面文件中应保留清晰的页面根节点或画布根节点，便于后续把需求说明作为独立文档层挂在页面之后。
11. 如果页面使用固定视口壳层或 `overflow-hidden` 布局，请保留一个可被外层纵向容器包裹的主页面根节点，不要把全部结构都锁死在不可扩展的裁切层里。
12. 输出的页面代码应便于后续分析，并能直接补充页面内需求说明区。
13. 如果页面未来要补需求说明，优先保留一个可识别的页面级主容器；不要把说明只能挂进抽屉、蒙版或固定浮层。
14. 如果未来会把需求说明直接写回页面，默认允许说明区使用 `data-spec-*` 属性与 `PROTO_SPEC` 注释标记，不要预先占用这些命名作为无关业务字段。

如果页面属于工作台、向导、报告、地图、平板断点等特殊形态，请仍然选择一个最接近的主页面类型和主页面形态进行组织，并显式保留其特有模块。
如果页面存在复杂抽屉或弹窗，请继续追加本文件中的 Drawer And Modal Rule。
```

## 2. 移动端版

适合 H5、移动端工作台、移动端详情、移动端表单、移动端工具页。

```text
为兼容 prototype-spec-annotator，请按移动端页面生成原型，并遵守以下约束：

1. 输出页面级原型代码文件，不要只输出零散组件。
2. 主页面形态固定为：移动端。
3. 主页面类型只选择一个：列表页 / 详情页 / 表单页 / 工具页。
4. 页面必须保留清晰的移动端结构，包括：顶部导航、返回路径、页面标题、主内容区、底部固定操作区或底部导航区。
5. 如果存在弹层，必须明确是全屏弹层、底部弹层还是弹窗，并保留可识别的打开与关闭入口。
6. 卡片列表、分段切换、吸底按钮、底部 Tab、操作菜单等关键模块应直接出现在页面代码中。
7. 字段名、按钮文案、状态文案、分栏标题、弹层标题必须直接写在模板中，不要只藏在配置里。
8. 页面应显式保留弱网态、加载中、空状态、失败态或无权限态中的至少一种可见结构。
9. 如果页面有长内容滚动，请保留固定头部、固定底部或关键操作入口，便于后续补充返回路径和显示隐藏规则。
10. 如果页面使用整页固定高度或裁切布局，请保留清晰的移动端页面根节点，便于后续把需求说明作为文档层挂在页面之后。
11. 不要把主要页面结构全部拆进不可见子组件；页面文件本身必须能看出主任务和主要交互链路。
12. 不要把未来的需求说明区域预留成覆盖式底部弹层；优先保留正文流末尾或可单独滚动的外层容器。
13. 如果后续会给需求说明加显隐控制，请允许说明区根节点使用 `data-spec-visible` 与 `hidden`，不要让业务样式强行覆盖这类属性行为。

该页面后续需要被分析并补充需求说明，因此请确保返回路径、底部操作、弹层层级、状态反馈都能从代码中直接识别。
如果页面存在复杂抽屉或弹窗，请继续追加本文件中的 Drawer And Modal Rule。
```

## 3. 大屏版

适合监控大屏、数据驾驶舱、运营看板、态势感知页面。

```text
为兼容 prototype-spec-annotator，请按大屏页面生成原型，并遵守以下约束：

1. 输出页面级原型代码文件，不要只输出零散图表组件。
2. 主页面形态固定为：大屏。
3. 页面类型优先选择最接近的一类，通常为列表页 / 详情页 / 工具页之一。
4. 页面必须保留可识别的数据模块分区，包括：标题区、时间范围区、指标卡区、图表区 / 地图区 / 告警区。
5. 数据更新时间、刷新入口、自动刷新提示、轮播区或联动区应直接出现在页面代码中。
6. 图表容器、模块标题、指标名称、告警名称、筛选名称必须直接出现在模板中，不要只藏在数据配置中。
7. 如果存在多块图表联动、钻取、高亮、告警闪烁、离线状态，请保留可见入口或状态区块。
8. 页面应能直接识别出时间范围切换、刷新周期、异常数据提示、空图表状态等结构。
9. 不要只生成纯视觉拼贴式大屏；页面结构必须能支撑后续补充数据规则和状态规则。
10. 输出代码应便于后续分析并补充页面内需求说明区。
11. 如果存在全屏画布或地图层，不要把未来说明区预留在悬浮覆盖层中；应保留画布外部或下方的安全说明区域。
12. 如果后续会把说明区直接挂在页面内，允许说明区使用独立的 `data-spec-*` 标记，不要让全局样式或脚本依赖这些属性名。
如果页面存在复杂抽屉或弹窗，请继续追加本文件中的 Drawer And Modal Rule。
```

## 4. AIGC 版

适合 AI 工作台、Agent 页面、对话式生成页、内容生成页。

```text
为兼容 prototype-spec-annotator，请按 AIGC 页面生成原型，并遵守以下约束：

1. 输出页面级原型代码文件，不要只输出零散组件。
2. 主页面形态固定为：AIGC。
3. 页面类型优先选择最接近的一类，通常为工具页 / 详情页 / 表单页之一。
4. 页面必须保留可识别的输入到输出主链路，包括：输入区、参数区、生成触发区、生成状态区、结果区。
5. 如果是对话式页面，必须保留消息区、输入框、发送入口、停止生成或重新生成入口。
6. 如果是工作台式页面，必须保留提示词输入、参数面板、结果卡片、引用来源区、复制 / 导出 / 插入后续流程等入口。
7. 输入限制、按钮文案、状态文案、结果标题、引用标题必须直接出现在模板中，不要只藏在抽象配置里。
8. 页面应显式保留生成中、失败、重试、空结果、部分输出或多版本结果中的至少一种可见结构。
9. 如果存在附件上传、上下文引用、历史记录、版本切换，请保留明确入口和结构容器。
10. 输出代码应便于后续分析，并能直接补充 AIGC 页面专属的需求说明规则。
11. 如果页面采用工作台式 `h-screen` 壳层，请让页面外层能安全承接后续说明区，不要只能通过右侧助手面板或底部覆盖层展示说明。
12. 如果页面后续会给说明区提供“默认隐藏 / 手动展开”能力，请在说明区头部保留可放置切换按钮的标题层，不要把标题层和正文彻底耦死。
如果页面存在复杂抽屉或弹窗，请继续追加本文件中的 Drawer And Modal Rule。
```

## 5. Selection Guide

优先这样选：

- 普通后台 / B 端页面：使用“通用版”
- 手机 / H5 页面：使用“移动端版”
- 监控或看板页面：使用“大屏版”
- AI 工作台或对话页面：使用“AIGC 版”

如果页面属于平板断点、工作台、向导、报告、地图等特殊形态，优先选最接近的主页面形态版本，再在原始提示词里补一段该形态的额外约束。
如果页面中还有复杂抽屉、弹窗或二级流程，统一继续追加 `Drawer And Modal Rule`。
