# 产品经理标注写作规范

## 1. 标注目标

原型标注不是解释代码，而是帮助评审人员理解：

1. 当前页面或组件的业务作用是什么；
2. 用户在什么场景下使用；
3. 用户操作后系统发生什么；
4. 成功、失败、空状态、异常状态如何反馈；
5. 是否涉及权限、规则、AI、数据、转化或页面跳转；
6. 哪些内容需要后续确认。

## 2. 默认写作结构

未启用 Page Specs Lite 时，`P` 页面级说明标注必须包含：

### 页面功能介绍

说明页面面向谁、解决什么核心任务。

### 核心内容

列出页面中的核心对象、关键模块、主要信息区和关键状态。

### 业务流程

说明用户进入页面、完成任务、进入下一步或返回上级页面的业务路径。

当页面流程包含多个步骤或分支时，可在本小节末尾追加简短 `flowchart LR`。

### 主要操作

列出页面上的关键操作、二级界面入口和高风险操作。

### 待确认

列出仍需要产品、业务或研发确认的问题。

启用 Page Specs Lite 时，`P` 页面级说明标注直接展示 `prototype-annotator/specs/current/<pageKey>.md` 整篇 Markdown，不在 `annotations.json` 中维护上述摘要正文。元素级产品标注建议包含：

### 业务含义

说明这个页面、组件、按钮或状态在业务流程中的作用。

### 使用场景

说明用户什么时候会用它。

### 交互规则

说明用户如何操作，系统如何响应。

### 状态与异常

说明加载、成功、失败、空状态、禁用状态等。

当标注涉及明确状态枚举或状态流转时，可追加 `stateDiagram-v2`；当标注涉及规则、权限、AI 或二级界面分支时，可追加简短 `flowchart TD`。普通字段、静态组件和一步操作不要强行加图。

### 待确认

列出仍需要产品、业务或研发确认的问题。

## 3. 禁用表达

在 `product-review` 模式下，禁止默认出现以下表达：

- 以源码为准
- 以组件 state 为准
- store action
- 本地 state
- mock 数据
- selector
- handler
- Zustand
- src/pages
- src/store
- 组件内部校验
- API.createApp
- openDrawer
- form.checkValidity()
- confirm(...)
- showToast
- 函数逻辑
- 源码
- 该元素
- 本条由本地规则草稿生成
- 待补：xxx
- 具体规则后续确认

## 4. 推荐表达方式

### 错误写法

字段含义与校验以 src/pages/portal/Home.tsx 组件 state 为准。

### 正确写法

用户输入关键词后，系统根据关键词筛选当前资源列表；关键词为空时恢复默认推荐列表。

### 错误写法

提交时调用 store action 持久化到 mock 数据。

### 正确写法

用户点击提交后，系统保存当前填写内容，并进入下一步审核流程。

### 错误写法

变更后触发本地 state 更新。

### 正确写法

用户修改筛选条件后，页面应立即刷新列表结果，并保留当前筛选条件。

## 5. 写作风格

1. 使用产品经理语言。
2. 优先解释业务含义，而不是实现方式。
3. 每条标注控制在 3-5 个小节。
4. 不要大段复制 PRD。
5. 不要把标注写成代码说明。
6. 对不确定内容使用「待确认」，但不要泛滥。
7. 标注要能帮助前端、设计、测试、客户理解页面。
