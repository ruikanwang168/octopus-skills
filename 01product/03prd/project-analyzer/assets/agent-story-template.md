# Agent故事标准格式

## 格式说明

Agent故事用于描述每个AI Agent在系统中的角色定位、任务场景和能力需求。标准格式如下：

```
## [Agent名称] Agent 模型故事

### Agent故事1: [故事标题]

#### 场景描述
:::info
当[描述具体场景条件]时，我需要[Agent需要完成的任务]。
:::

#### 模型故事
```markdown
作为 [Agent角色名称]
在 [任务场景描述]时，
为了 [完成的目标]，
我需要：

【上下文信息】
- 上下文信息1
- 上下文信息2
- 上下文信息3

【能力支持】
- 能力1
- 能力2
- 能力3

【工具清单】
1. [工具名称1](参数说明)
   - 用途: 工具的主要功能
   - 何时使用: 使用场景描述

2. [工具名称2](参数说明)
   - 用途: 工具的主要功能
   - 何时使用: 使用场景描述

【约束条件】
- 约束1
- 约束2
- 约束3
```

#### 为什么需要这些
- 理由1
- 理由2
- 理由3
```

## 关键要素说明

### 1. 场景描述
- 使用 `:::info` 格式突出显示
- 清晰描述触发条件：什么时候、什么情况下
- 简洁说明任务目标

### 2. 模型故事
使用Markdown代码块，包含以下部分：

**上下文信息（Context）**
- Agent执行任务时需要的背景信息
- 包括：用户输入、历史记录、系统状态、时间等
- 确保信息完整性和准确性

**能力支持（Capabilities）**
- Agent需要具备的核心能力
- 包括：理解能力、判断能力、生成能力等
- 越具体越好，便于开发评估

**工具清单（Tools）**
- Agent可以调用的工具或API
- 每个工具包含：名称、参数、用途、使用时机
- 工具调用应该具体明确

**约束条件（Constraints）**
- Agent必须遵守的限制
- 包括：输出格式、时间限制、禁止行为等
- 确保系统稳定性和可控性

### 3. 为什么需要这些
- 解释设计决策的理由
- 帮助团队理解Agent的设计逻辑
- 便于后续评估和优化

## 示例

### Coordinator Agent示例

```markdown
## Coordinator Agent 模型故事

### 模型故事1：识别用户意图

#### 场景描述
:::info
当用户发送第一条消息时，我需要快速判断这是简单对话、不当请求还是研究请求。
:::

#### 模型故事
```markdown
作为 Coordinator Agent
在 接收到用户首次输入时
为了 正确分类并采取对应行动(直接回复/拒绝/移交)
我需要:

【上下文信息】
- 用户的原始输入文本
- 当前对话的语言locale(如果能检测到)
- 系统当前时间

【能力支持】
- 多语言理解能力(支持中英文等主流语言)
- 意图分类能力(三分类:问候/不当/研究)
- 工具调用能力(能主动调用handoff_to_planner工具)

【工具清单】
1. handoff_to_planner(research topic, locale)
   - 用途:将研究任务移交给Planner
   - 何时使用:识别到研究请求时

【约束条件】
- 不能尝试直接回答研究问题
- 对于研究请求必须调用工具，不能文本回复
- 需要在3秒内完成意图识别
```

#### 为什么需要这些
- 无历史对话：首次对话时没有上下文，需要单独依赖输入文本
- 快速决策：不需要深度思考，只需要快速分类
- 强制动作：通过工具绑定确保模型执行正确动作而非自由发挥
```

## 常见Agent类型

### 1. Coordinator Agent（协调器）
- 职责：意图识别、任务分发、流程控制
- 常见场景：首次交互、状态判断、Agent切换

### 2. Planner Agent（规划器）
- 职责：任务分解、计划生成、上下文评估
- 常见场景：研究计划、工作流设计、步骤拆解

### 3. Researcher Agent（研究员）
- 职责：信息搜索、内容爬取、数据整理
- 常见场景：网络搜索、知识库检索、页面爬取

### 4. Coder Agent（编码器）
- 职责：数据处理、代码执行、结果计算
- 常见场景：API调用、数据分析、计算任务

### 5. Writer Agent（写作器）
- 职责：内容生成、格式化、风格调整
- 常见场景：报告生成、摘要创作、内容润色

## 编写技巧

### 1. 场景要具体
- ❌ 错误："处理用户请求"
- ✅ 正确："当用户发送第一条消息时"

### 2. 约束要明确
- ❌ 错误："要快一点"
- ✅ 正确："需要在3秒内完成"

### 3. 工具要实用
- ❌ 错误："调用各种工具"
- ✅ 正确："handoff_to_planner(research topic, locale)"

### 4. 理由要充分
- 解释设计背后的思考，帮助团队理解决策逻辑
