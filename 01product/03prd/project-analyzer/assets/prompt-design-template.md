# 提示词设计策略标准格式

## 格式说明

提示词设计策略用于指导如何为每个Agent设计有效的提示词，确保Agent能够稳定、准确地完成预期任务。标准格式如下：

```
## [Agent名称] 提示词设计策略

### 角色定位
- 角色名称：[Agent的角色定义]
- 核心职责：[Agent的主要任务和责任边界]
- 定位描述：[详细描述该Agent在整个系统中的作用]

### 核心挑战
- 挑战1：[描述第一个核心技术或业务挑战]
- 挑战2：[描述第二个核心挑战]
- 挑战3：[描述第三个核心挑战]

### 设计策略
#### 策略1：[策略名称]
- 为什么这样设计：[为什么采用这个策略]
- 实施方法：[具体如何实现]
- 预期效果：[达到什么目的]

#### 策略2：[策略名称]
- 为什么这样设计：[为什么采用这个策略]
- 实施方法：[具体如何实现]
- 预期效果：[达到什么目的]

#### 策略3：[策略名称]
- 为什么这样设计：[为什么采用这个策略]
- 实施方法：[具体如何实现]
- 预期效果：[达到什么目的]

## 关键要素说明

### 1. 角色定位
**角色名称**：清晰定义Agent的身份
- ❌ 模糊："智能助手"
- ✅ 具体："研究任务规划器"

**核心职责**：明确职责边界
- 说明Agent负责什么、不负责什么
- 与其他Agent的职责区分

**定位描述**：详细作用
- 在系统整体架构中的位置
- 与其他Agent的协作关系

### 2. 核心挑战
识别并描述3-5个核心挑战：

**技术挑战**
- 模型的理解能力限制
- 上下文窗口的约束
- 工具调用的准确性

**业务挑战**
- 用户意图的模糊性
- 任务分解的合理性
- 输出质量的稳定性

**系统挑战**
- 与其他Agent的协作
- 异常情况的处理
- 性能和成本的平衡

### 3. 设计策略
针对每个挑战提出具体策略：

**结构化设计**
- 明确的提示词结构（角色、任务、工具、约束等）
- 清晰的输出格式要求（JSON、Markdown等）
- 完善的示例和说明

**思维链引导**
- 引导模型进行逐步推理
- 使用"思考-行动-输出"的模式
- 避免模型直接跳过思考

**工具绑定**
- 明确哪些情况下必须调用工具
- 提供工具参数的详细说明
- 给出工具使用的示例

**约束强化**
- 在多个地方重复重要约束
- 使用加粗、列表等方式突出
- 通过示例演示正确和错误的用法

**Few-shot示例**
- 提供2-3个完整的输入-输出示例
- 覆盖常见场景和边界情况
- 确保示例的真实性和代表性

### 4. 完整提示词
提供可以直接使用的提示词文本，包含：

**角色定义部分**
- 清晰的角色定位
- 职责范围说明

**任务目标部分**
- 详细的任务描述
- 明确的执行步骤

**工具使用部分**
- 工具列表和说明
- 参数规范
- 使用场景

**输出要求部分**
- 格式要求（JSON schema、Markdown结构等）
- 内容要求（必须包含的信息）
- 质量标准

**约束条件部分**
- 必须遵守的规则
- 禁止的行为
- 边界限制

**示例部分**
- 至少2-3个完整示例
- 覆盖不同场景
- 展示正确的输出格式

### 5. 关键输出控制
确保输出质量的控制机制：

**输出格式控制**
- JSON schema验证
- Markdown结构要求
- 字段必填/选填说明

**内容质量控制**
- 准确性检查（事实、数据、引用）
- 完整性检查（是否遗漏必要信息）
- 一致性检查（前后是否矛盾）

**边界情况处理**
- 输入不完整时的处理
- 工具调用失败时的处理
- 超时或异常时的处理

## 示例：Planner Agent提示词设计策略

```markdown
## Planner Agent 提示词设计策略

### 角色定位
- 角色名称：研究任务规划器
- 核心职责：将用户的研究主题分解为可执行的步骤计划
- 定位描述：Planner Agent是研究流程的"大脑"，负责理解用户意图、评估信息需求、生成结构化的研究计划，是后续Agent执行的蓝图

### 核心挑战
- 挑战1：用户研究主题往往模糊不清，需要通过上下文理解并提取关键维度
- 挑战2：需要在信息全面性和执行效率之间找到平衡，步骤过多导致效率低，过少导致信息不全面
- 挑战3：步骤的描述必须足够具体，才能让Researcher Agent准确执行

### 设计策略

#### 策略1：结构化输出设计
- 为什么这样设计：使用固定的JSON schema确保输出格式的一致性，便于程序解析和后续处理
- 实施方法：
  - 定义严格的JSON schema，包含locale、thought、title、steps等字段
  - 每个步骤包含title、description、need_search等子字段
  - 使用示例演示正确的JSON格式
- 预期效果：输出格式稳定，程序处理可靠

#### 策略2：思维链引导
- 为什么这样设计：引导模型在输出最终计划前先进行思考，避免直接跳过推理
- 实施方法：
  - 在thought字段中要求模型"重述用户需求，说明为什么需要这些步骤"
  - 通过few-shot示例展示thought的写法
  - 强调thought必须反映真实的推理过程
- 预期效果：提高计划的合理性和可解释性

#### 策略3：8维度框架引导
- 为什么这样设计：提供标准的分析维度，避免模型遗漏重要方面
- 实施方法：
  - 列出8个参考维度：历史背景、当前状况、未来趋势、利益相关方、定量数据、定性信息、对比分析、风险挑战
  - 要求模型根据主题选择最重要的3-5个维度
  - 强调不是每个主题都需要所有维度
- 预期效果：计划的全面性和针对性提升

#### 策略4：步骤类型区分
- 为什么这样设计：明确区分需要搜索的步骤和不需要搜索的步骤，提高执行效率
- 实施方法：
  - 每个步骤标记need_search字段（true/false）
  - true：需要调用web_search、crawl等工具
  - false：只需要数据处理、计算等
  - 提供两者的示例对比
- 预期效果：避免不必要的工具调用，提高效率

### 完整提示词
```markdown
你是一个研究任务规划器，负责将用户的研究主题分解为可执行的步骤计划。

你的核心能力：
- 理解用户的研究需求和意图
- 评估信息需求，判断是否需要搜索
- 将复杂主题分解为逻辑清晰的步骤
- 为每个步骤定义明确的数据需求

当收到研究主题时，你需要：

1. 评估上下文充足性：基于当前信息判断是否已经可以直接回答用户问题
2. 如果不充足，进行任务分解：将研究主题分解为3-5个可执行的步骤
3. 为每个步骤定义：标题、描述、是否需要搜索

任务分解时，考虑以下维度（选择最重要的3-5个）：
- 历史背景（如果需要）
- 当前状况（主要）
- 未来趋势（如果相关）
- 利益相关方（如果适用）
- 定量数据（数字、统计）
- 定性信息（观点、案例）
- 对比分析（如果需要）
- 风险和挑战（如果重要）

输出格式要求（严格JSON格式）：
```json
{
  "locale": "zh-CN",
  "has_enough_context": false,
  "thought": "重述用户需求，说明为什么需要这些步骤",
  "title": "研究计划标题",
  "steps": [
    {
      "title": "步骤标题",
      "description": "具体要收集什么数据，从哪些角度",
      "need_search": true,
      "step_type": "research"
    }
  ]
}
```

约束条件：
- 步骤数量：不超过max_step_num（通常为3或5）
- 每步要具体：不能说"收集信息"，要说"收集2024年销量数据"
- 步骤互不重叠：不同步骤聚焦不同方面
- 有逻辑顺序：通常先基础再深入
- 不包含总结步骤：不要加"汇总分析"这样的步骤

示例：

**输入**："分析量子计算对密码学的影响"

### 关键输出控制

#### 输出格式控制
- 必须是有效的JSON格式，可以通过JSON解析器
- 必须包含所有必需字段：locale、has_enough_context、thought、title、steps
- steps数组中的每个对象必须包含：title、description、need_search

#### 内容质量控制
- thought必须真实反映推理过程，不能是空泛的描述
- 每个步骤的description必须具体明确，不能笼统
- 步骤之间要有逻辑关系，形成递进结构
- 如果has_enough_context=true，steps必须为空数组

#### 边界情况处理
- 当主题过于模糊时，可以在thought中说明并建议澄清，但仍需生成基础计划
- 当主题超出专业领域时，可以生成保守的计划（通用性步骤）
- 当达到最大步骤数但仍未覆盖完整需求时，可以在最后一步注明"可能需要后续补充"
```

## 编写技巧总结

### 1. 策略要可落地
- ❌ 抽象："提高理解能力"
- ✅ 具体："提供few-shot示例，覆盖常见场景"

### 2. 约束要强化
- 在角色定义、任务目标、约束条件等多个位置重复
- 使用不同的表达方式强调
- 通过示例演示正确和错误的用法

### 3. 示例要真实
- ❌ 虚构的输入输出
- ✅ 真实的用户请求和Agent响应
- 示例应具有代表性，能覆盖主要使用场景

### 4. 格式要严格
- 明确JSON schema、Markdown结构等
- 使用代码块格式化提示词
- 提供格式验证方法

### 5. 迭代优化
- 根据实际使用反馈调整提示词
- 记录常见的错误模式和修复方法
- 持续完善示例和约束
