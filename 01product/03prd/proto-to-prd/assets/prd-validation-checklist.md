# PRD 自检清单 (Validation Checklist)

在交付 PRD 前，请对照以下各项进行严格检查，确保文档质量符合工程化执行标准。

---

## 1. 结构完整性检查
- [ ] 是否根据应用类型（AI/传统）选择了正确的模板？
- [ ] 是否通过引用或包含的方式使用了 `base-metadata.md`，并按平台选择了 `web-library.md` / `mobile-library.md`？
- [ ] 目录索引是否与实际内容一致？

## 2. 需求覆盖度检查
- [ ] **功能覆盖索引**：是否在 PRD 中新增了《功能覆盖索引（Coverage Index）》并包含 Step 2 的 `pages` 表？
- [ ] **无缺失**：Coverage Index 中是否不存在 `❌缺失`？如信息不足，是否以 `[待确认]` + `open_questions` 闭环而不是遗漏？
- [ ] **全模块覆盖**：Step 2 `pages` 表识别出的所有页面/模块是否都在对应章节有详细描述，并能在 Coverage Index 中通过 `PRD章节定位` 精确跳转？

## 3. UI 与交互描述检查
- [ ] **ASCII 原型**：每个页面是否都有清晰的 ASCII 线框图？
- [ ] **组件匹配**：线框图是否符合 `web-library.md` / `mobile-library.md` 中的标准组件规范？
- [ ] **交互逻辑**：是否清晰描述了按钮点击、页面跳转、状态切换逻辑？

## 4. 业务规则、流程图与一致性检查（评审重点）
- [ ] **流程图最小集**：是否至少包含 1 张端到端主流程 Mermaid Flowchart？
- [ ] **[新增] 逻辑一致性（Vibe Coding 专项）**：是否检查并消除了原型中的“死胡同”交互或由于多次对话修改导致的过时组件引用？
- [ ] **[新增] 业务意图溯源**：是否在每个页面章节记录了 PM 设计该交互的原始意图或核心解决问题？
- [ ] **异常分支**：关键流程图是否包含至少 1 条异常/失败分支，并在用户故事 AC 中以 EARS `Unwanted Behavior` 对齐？
- [ ] **逐段解释**：每张 Mermaid 图是否都附带逐段文字解释，逐条对应节点/分支？
- [ ] **工程化数据字典（P0/P1/P2）**：
  - [ ] P0 必填（字段名/含义/类型/必填/来源证据）是否齐全？
  - [ ] P1（校验/长度/默认值）缺失时是否统一标注 `[待确认: 缺失项 | 影响 | 建议负责人]` 且不脑补？
  - [ ] 数据字典是否包含“来源/证据”列（或等价证据说明）？

## 5. Evidence 与客观性检查（防脑补）
- [ ] **页面 Evidence 汇总**：每个页面/模块章节是否包含 Evidence 小节（code/a11y/screenshot/old_prd/unknown）？
- [ ] **混合输入证据**：对关键结论是否尽量提供 ≥2 条 Evidence（例如 code+a11y / code+screenshot）？
- [ ] **缺失处理**：缺失或模糊的信息是否标明为 `[待确认]` 并出现在 `open_questions` 表中？

## 6. 升级迭代专项检查 (针对迭代场景)
- [ ] **变更记录**：文档开头是否已包含《版本变更记录表》？
- [ ] **增量标识**：新增内容是否标注了 `[NEW]`，修改内容是否标注了 `[vNext]`？
- [ ] **Diff 表**：是否输出逐项 Diff 表（粒度：页面/字段/规则/流程；影响面：DB/API/权限/埋点/测试）？
- [ ] **关联影响**：是否识别并描述了本次变更对存量功能的影响？
- [x] **逻辑一致性**：更新后的逻辑是否与未改动的存量逻辑保持闭环？

## 7. 业务深度与“去 Mock 化”专项检查 (P0)
- [ ] **排除 Mock 数据**：是否已剔除原型中的测试数据（如 "test_user"、"123456"）并转为业务规则或示例值？
- [ ] **接口脱钩**：是否已剔除原型中的临时接口路径（如 `/api/mock/xxx`）并转为业务意图描述？
- [ ] **字段业务化**：数据字典中的字段名是否为业务语义名，而非代码中的 `var_abc` 等技术命名？
