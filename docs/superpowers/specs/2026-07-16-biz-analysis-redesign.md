# Skill 需求文档：biz-analysis 重构设计

## 结论

- 推荐动作：更新现有 Skill。
- 推荐名称：`biz-analysis`。
- 推荐执行约束模式：检查点型
- 选择依据：需求澄清允许根据上下文灵活追问，但项目分类、16 点覆盖判断、最终共识确认和文档生成是不可跳过的检查点；任务不包含不可逆外部操作，不需要 SOP 型约束。
- 推荐资源：`SKILL.md`、`references/analysis-framework.md`、`references/document-template.md`、`agents/openai.yaml`。
- 目标路径：`01product/01demand/biz-analysis`。

## 这个 Skill 只解决什么问题

将用户模糊、零散或直接以功能表达的业务诉求，梳理为覆盖 16 个核心考量点的业务需求基线。产物供业务确认，并为后续 PRD、原型、数据库设计和研发任务提供输入，但不直接完成这些下游设计。

## 触发场景

- 用户可能会说：“帮我梳理一个新的业务系统需求。”
- 用户可能会说：“现有系统要加一个功能，帮我把需求和影响范围理清楚。”
- 用户可能会说：“我要做一个处理 Excel 的小工具，先帮我把需求边界确认下来。”
- 用户需要梳理审批、工单、订单、台账、运营后台、内部工具或 SaaS 模块的业务需求。

## 不适用场景

- 不处理：用户只要求制作页面原型、视觉稿或交互说明的任务。
- 用户已经有确认完成的需求基线，只要求编写完整 PRD、接口、数据库或技术方案时，不触发本 Skill。
- 用户只要求修改代码、数据库、配置或原型文件时，不触发本 Skill。
- 不生成竞品分析、市场研究或营销方案。

## 任务契约

### 输入

- 必需输入：业务场景、当前问题或希望发生的业务变化；最少提供其中一项即可启动。
- 可选输入：现有需求文档、流程图、原型、截图、操作说明、系统地址、业务数据和历史方案。
- 输入缺失时：先根据已有信息形成“当前理解”，再追问最影响业务判断的 2 至 4 个问题；非关键缺口标记为待确认，不阻塞后续澄清。

### 输出

- 输出形式：经用户确认后创建或更新 Markdown 文件 `需求梳理文档.md`。
- 输出框架：认知层、边界层、交付层、防御层，共 16 个考量点，并按全新产品、增量迭代、小工具三类项目剪裁。
- 输出必须包含：项目类型、必填项结论、适用条件项结论、当前推测、待确认问题，以及能够追溯到痛点、角色、流程或防御要求的功能清单。
- 输出不得出现：页面布局、按钮、弹窗、导航、列表列项、表单字段、字段校验、接口定义、数据库字段、技术架构、监控实现方案等 PRD、原型或技术设计细节。

### 红线

- 不把 16 个考量点机械改写成长问卷。
- 不把用户提出的功能直接视为真实需求；必须核对触发场景、当前做法、真正障碍、业务影响和期望结果。
- 不重复询问已有材料能够证明的内容。
- 不主动深挖没有适用信号的条件填写项。
- 不编造业务规模、性能数字、角色权限、业务状态、外部依赖或历史数据策略。
- 不因功能清单需要分级而询问页面菜单或交互结构。

### 验收标准

- 必填项均有已确认结论，或有具体待确认问题及其影响。
- 条件填写项明确标记适用、不适用或待判断。
- 每轮只提出 2 至 4 个能够改变业务判断的问题。
- 关键需求形成“表面诉求 → 触发场景 → 当前做法 → 真正障碍 → 业务影响 → 期望结果 → 必要能力”的因果链。
- 功能清单可作为后续开发任务拆分的输入，但不包含原型和技术实现细节。
- 全新产品、增量迭代和小工具使用正确的深度与表格结构。
- 用户确认最终共识摘要后才生成正式文档。

## 执行约束设计

- 推荐模式：检查点型
- 模型可自主决定：每轮追问的具体问题、同一问题覆盖哪些考量点、考量点的澄清顺序、章节合并方式和摘要表达方式。
- 必须遵守：项目类型确认、材料优先提取、16 点适用性判断、真实需求因果链、覆盖状态检查、最终共识确认和文档生成门槛。
- 升级条件：不涉及不可逆外部操作，不需要升级为 SOP 型；若用户切换到发布、数据迁移或权限变更等任务，应退出本 Skill 并使用对应能力。

### 必经检查点

1. 项目路由检查点：进入条件为业务场景基本明确；必须产物为已确认的项目类型；退出条件为已选定全新产品、增量迭代或小工具，并读取对应适用性规则。
2. 真实需求检查点：进入条件为用户提出痛点、功能或方案；必须产物为关键诉求的业务因果链；退出条件为核心痛点和关键功能均能追溯到真实场景、影响和期望结果。
3. 需求基线检查点：进入条件为主要业务链路已经形成；必须产物为 16 点覆盖状态；退出条件为必填项已闭环、条件项已判断、冲突和关键缺口已暴露。
4. 交付检查点：进入条件为需求基线已形成；必须产物为最终共识摘要；退出条件为用户明确确认摘要并同意生成文档。

### SOP 与异常处理

不适用。冲突资料、关键规则缺失或目标范围不明确时，暂停文档生成并回到澄清阶段；不自行裁决或补造结论。

### 阶段映射

不适用。本 Skill 全程采用检查点型约束。

## 核心流程

1. 识别用户描述和材料中的已有证据，复述当前理解。
2. 判断项目类型并请用户确认。
3. 读取 `references/analysis-framework.md`，建立 16 点覆盖表，状态为已确认、当前推测、待确认或不适用。
4. 优先处理矛盾、必填缺口和高风险问题；每轮只问 2 至 4 个问题。
5. 对关键诉求执行真实需求因果链和必要性检验，直到真实障碍、业务影响和必要能力明确。
6. 每轮更新短摘要，不向用户展示冗长检查表。
7. 达到需求基线后展示完整覆盖状态和最终共识摘要。
8. 用户确认后读取 `references/document-template.md`，按项目类型生成 `需求梳理文档.md`。

|---|---|---|---|---|---|---|---|

| 动作点 | 输入 | 处理规则 | 输出 | 异常处理 |
|---|---|---|---|---|

功能能力分层不是页面菜单。每个功能项必须能追溯到痛点、角色、流程节点或防御要求。

## references 设计

- `references/analysis-framework.md`：项目类型确认后读取，用于 16 点适用性判断、真实需求挖掘、追问准入、每项分析边界和停止条件。
- `references/document-template.md`：最终共识摘要得到用户确认后读取，用于把已确认结论组织为正式文档。
- 不读取 references 的情况：首次响应只需理解场景和确认项目类型时，不读取文档模板。

## 行为验证场景

1. 全新产品：用户要求建设会员运营系统。Skill 应先挖掘业务目标和真实障碍，再覆盖全量业务结构；不得询问页面布局、按钮或列表列项。
2. 增量迭代：用户要求给现有退款流程增加审批。Skill 应聚焦新老流程差异、状态和权限变化、异常、依赖及老数据；不得重梳整个系统或设计弹窗。
3. 小工具：用户要求制作 Excel 对账工具。Skill 应聚焦痛点、方案、边界、输入处理输出和异常；不得强行补业务实体、状态、权限或监控。

失败判定：主动追问原型细节；把表面功能直接当作真实需求；机械遍历 16 点；功能项无法追溯；进入技术实现层；项目类型剪裁错误。

## 容易跑偏的点

- 将“深挖真实需求”误写成机械的“五个为什么”。
- 为了让功能清单可开发而进入页面、接口或数据库设计。
- 将所有条件填写项都当作必填，造成过度询问。
- 只问用户想要什么，不核对当前做法和不解决时的业务后果。
- 增量需求复述全量系统，小工具套用完整产品结构。
